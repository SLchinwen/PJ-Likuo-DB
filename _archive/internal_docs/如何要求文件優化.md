# 如何要求文件優化

與協作者或 AI 請求調整文件時，可依需求選用下列方式，較易得到精準結果。

## 1. 指定範圍與目標

- **範圍**：哪一份（或哪一段）文件，例如「`docs/index.md` 的雛型檢視區塊」。
- **目標**：希望達成的效果，例如「版面簡潔、重複訊息合併」、「表格易讀」、「標題層級一致」。

範例：「請優化 `docs/index.md` 第 5 節雛型檢視的排版，把多個尚未發布的項目合併成一個區塊或表格，讓已發布的 CUS100 更突出。」

## 2. 常見優化類型

| 類型 | 可這樣要求 |
| :--- | :--- |
| 減少重複 | 「把重複的提示合併成一個區塊／表格」 |
| 版面簡潔 | 「用表格取代多段相同格式的列舉」 |
| 層級與結構 | 「標題改為 H2／H3，並補上目錄」 |
| 用語一致 | 「全文『雛型／雛形』統一為『雛型』」 |
| 連結與路徑 | 「雛型連結改為相對路徑／統一命名」 |

## 3. 附上截圖或說明

- 若版面有問題：可附**截圖**或說明「目前長怎樣、希望長怎樣」。
- 若為 MkDocs 頁面：可註明「這是 MkDocs 產生的頁面」，方便保留 admonition（??? success / note）、表格等語法。

## 4. 對外 vs 內部文件

- **對外文件**（`docs/index.md`，即對外網站首頁）：不寫設計路徑、不寫狀態；雛型僅用連結。見 [文件治理規範](docs_治理規範.md)。
- **內部文件**（本目錄 `internal_docs/`）：可含路徑、狀態與撰寫指引。新增或優化時請依治理規範判斷放置位置。

## 5. 本次已做的優化（參考）

- **雛型檢視**：將多個「尚未發布」的 `??? tip` 合併為單一 `??? note`，內用表格列出 CUS110～CUS160，已發布的 CUS100 維持獨立 `??? success` 與連結。
- **說明文字**：精簡開頭說明與區塊內指引，避免重複。

之後若有新雛型發布，只要在專案總攬的程式清單表格「雛型」欄補上連結即可。
