# 新專案參考使用辦法｜整合 AI 開發

本文件彙整**公司規範、開發流程標準、資安規範、文件要求**，並經**整合 AI 開發**需求檢核，供**開新專案**時作為空白專案框架與預設治理文件之參考。可直接複製至新專案 Repo 之 `internal_docs/`，並依專案名稱、Repo 網址、角色名稱等替換後使用。

---

## 1. 適用情境與目的

| 項目 | 說明 |
| :--- | :--- |
| **適用對象** | 需求方持有 Repo、公司開發 TEAM 負責實作之 e化 專案；採用 **Cursor／AI 輔助**產出 PRD、雛型、文件時尤為適用。 |
| **目的** | 新專案立項時，依本辦法建立目錄結構、複製或引用治理文件，使公司規範、開發流程、資安與文件要求一致；並確保 **AI 產出**有明確依據與審核流程。 |
| **整合 AI 開發** | 本辦法已檢核：範本與 .cursorrules 可引導 AI 產出；PRD／雛型／文件有四項必備與資安建議；AI 產出須經需求方審核後才合併；單一需求來源（需求方 Repo）為對照基準。 |

---

## 2. 公司規範摘要

| 規範 | 要點 | 詳細文件（複製至新專案時對應路徑） |
| :--- | :--- | :--- |
| **公司專案標準｜需求與開發協作** | 需求方持有 Repo、授權開發方唯讀；開發方在**自有專案**實作（如 VS）；雙方**不同 Repo** 避免衝突；每週同步、客戶回饋由需求方通知。 | `internal_docs/公司專案標準規範_需求與開發協作.md` |
| **協作與治理** | 角色（需求方／開發方）、產物歸屬（01_Discovery、02_Analysis、docs、internal_docs、00_Templates）、分支與 PR、交付發布、**PRD 產出要求（四項必備）**、**開發要求（含資安）**、雛型設計階段資安對應。 | `internal_docs/協作與治理規範.md` |
| **程式編碼原則｜共用** | 模組代號、主程式代號（x00/x10/x20/x30）、權限表僅登錄主程式、子編碼由程式設計師編列、雛型檔名 `YYYY-MM-DD_prototype_<程式代號>.html`。 | `internal_docs/程式編碼原則_共用規範.md` |

新專案**必備**：至少複製或引用「公司專案標準規範」與「協作與治理規範」；程式編碼原則可依專案複製並改寫模組代號與百位數。

## 3. 開發流程標準摘要

| 階段 | 標準步驟 | 備註 |
| :--- | :--- | :--- |
| **需求產出** | 訪談 → 模組與主程式代碼 → PRD（四項必備）→ 雛型與對外文件 | 見 [文件治理規範](docs_治理規範.md) §6、[協作與治理規範](協作與治理規範.md) §2.4 |
| **授權與啟動** | 需求方授權 Repo 給開發人員 → 開發方建立自有專案 → 對齊需求方 Repo 版次／Tag | 開發方不於需求方 Repo 提交程式碼 |
| **開發與同步** | 每週討論進度與測試修正；測試與修正於開發方專案內完成 | 需求方可提供驗收情境或測試案例 |
| **變更與回饋** | 客戶回饋由需求方通知開發方；需求方更新 PRD／雛型後註明版次或變更摘要 | 需求方 Repo 為文件與發布單一依據 |
| **交付與發布** | 交付定義依專案約定；對外文件版次以需求方 Repo 為準；發布由需求方管理 | 可約定開發方協助建置／部署 |

**分支與 PR**：main 建議設 Branch protection，合併須經 PR；涉及 01_Discovery、02_Analysis、docs、雛型之 PR 建議由**需求方**審核。

## 4. 資安規範摘要

### 4.1 開發階段（開發方規劃與實作時）

依 [協作與治理規範](協作與治理規範.md) **§2.5 開發要求**：

| 類別 | 提醒要點 |
| :--- | :--- |
| 身分認證與登入 | 密碼不可明文；Session 逾時與登出；必要時 MFA |
| 授權與權限 | 依權限矩陣與情境、最小權限；前後端皆驗證，不得僅依前端隱藏 |
| 敏感資料 | 報關／個資之存取與顯示受控；日誌與除錯不記密碼或完整敏感欄位 |
| 傳輸 | HTTPS；敏感參數不置於 URL／Cookie；API 依要求加密 |
| 日誌與稽核 | 重要操作可追溯；日誌不記敏感內容；依公司或客戶規定保留 |
| 依循既有規範 | 依公司資安政策與客戶資安／個資要求；弱掃／滲透／合規預留時程 |

### 4.2 雛型設計階段（需求方或產出雛型時）

依 [文件治理規範](docs_治理規範.md) **§7 雛型設計與產出（含資安建議）**：

| 對應類別 | 雛型設計／生成時建議 |
| :--- | :--- |
| 授權與權限 | 雛型中區分角色示範（標示「僅 XX 角色可見」、選單／功能差異） |
| 敏感資料 | 敏感欄位以遮罩或假資料示範，不使用真實客戶／報關資料 |
| 登入與登出 | 保留登出示意；可註記逾時與 Session 由正式系統實作 |
| 傳輸與環境 | 註記「正式環境須 HTTPS」「實際送出須經後端驗證與加密」 |
| 對齊整體要求 | 參照協作與治理 §2.5；客戶／公司額外資安規定寫入 PRD 或雛型說明 |

## 5. 文件要求摘要

### 5.1 對外 vs 內部

| 對象 | 目錄 | 規則 |
| :--- | :--- | :--- |
| **使用者／客戶** | `docs/` | 僅專案總攬、程式清單、雛型**連結**、版次與發布日；**不寫**設計路徑、內部狀態、待規劃、開發流程細節 |
| **開發團隊** | `internal_docs/` | 程式編碼原則、協作與治理、文件治理、PRD 索引、提交步驟等；**不透過 MkDocs 對外發布** |

### 5.2 功能作業 PRD（四項必備）

產出或審核 PRD 時須齊備（見 [文件治理規範](docs_治理規範.md) §6、[協作與治理規範](協作與治理規範.md) §2.4）：

1. **模組與主程式代碼**：完整主程式代號清單（權限表用）
2. **程式 × 使用者角色權限矩陣表**
3. **對應情境說明**
4. **模組欄位名稱中英文對照表**（表單與報表欄位：中文、英文、縮寫、用途）

範本：`00_Templates/Tpl_PRD_功能作業.md`、`Tpl_權限矩陣表.md`、`Tpl_模組欄位名稱對照表.md`。

### 5.3 雛型設計與產出

- **命名**：`YYYY-MM-DD_prototype_<程式代號>.html`
- **放置**：分析期可置 `02_Analysis/`；發布給客戶複製至 `docs/prototype/`
- **資安**：依文件治理規範 §7（權限區分示範、敏感欄位遮罩、登出／逾時示意、HTTPS 與後端驗證註記）

## 6. 整合 AI 開發檢核結果

以下項目已與現行公司規範、開發流程、資安與文件要求對齊，**符合整合 AI 開發**之需求；開新專案時請一併納入。

| 檢核項 | 狀態 | 說明 |
| :--- | :--- | :--- |
| **單一需求來源** | ✅ | 需求方 Repo 為 PRD／雛型／文件之單一依據；開發方對齊其版次／Tag，利於 AI 與人工共同參照。 |
| **結構化範本** | ✅ | 00_Templates 提供 PRD、權限矩陣、欄位對照表範本；AI 產出時可明確引用範本格式，減少歧義。 |
| **.cursorrules 引用治理** | ✅ | .cursorrules 已引用協作與治理、文件治理、目錄與雛型命名；AI 產出會依相同規則歸檔與命名。 |
| **文件治理（對外/內分離）** | ✅ | 對外（docs）不暴露路徑與狀態；內部（internal_docs）可含路徑與指引；AI 產出時可依此判斷放置位置。 |
| **PRD 四項必備** | ✅ | 模組與主程式代碼、權限矩陣、情境說明、欄位對照表齊備後再提供工程方；AI 產出 PRD 時可依自檢表勾選。 |
| **資安貫穿需求與開發** | ✅ | 開發階段 §2.5、雛型階段 §7 已納入；AI 產出雛型或 PRD 時可依資安建議檢視。 |
| **AI 產出之審核歸屬** | ✅ | 涉及 01_Discovery、02_Analysis、docs、雛型之 PR 建議由需求方審核；AI 協助產出之文件仍須經人審核後合併。 |
| **版本紀錄與可追溯** | ✅ | 治理要求 YAML Frontmatter 與版本紀錄；利於 AI 與人工辨識版次與變更。 |
| **如何要求文件優化** | ✅ | internal_docs 已有「如何要求文件優化」，可引導與 AI 或協作者請求調整文件之撰寫方式。 |

**建議補強（可選）**：於 .cursorrules 或協作規範中明寫「**AI 輔助產出之 PRD、雛型、文件須經需求方（或專案負責人）審核後再合併至 main**」，以利新成員或新專案一致遵循。

## 7. 新專案套用步驟（空白專案框架）

### 7.1 目錄結構建議

```
Repo 根目錄/
├── 00_Templates/          # 範本庫（PRD、權限矩陣、欄位對照表等）
├── 01_Discovery/          # 訪談、客戶資料卡、需求展開
├── 02_Analysis/           # 分析、PRD、雛型規範、雛型 HTML
├── 03_Solution/           # 規格細化、技術設計（若採用）
├── docs/                  # 對外文件（專案總攬、雛型發布用、操作手冊）
│   └── prototype/         # 發布給客戶之雛型 HTML（可選）
├── internal_docs/          # 內部治理與開發團隊文件（不對外發布）
├── .cursorrules           # Cursor／AI 行為與引用治理（必備）
├── mkdocs.yml             # 若使用 MkDocs 對外建站
└── .github/workflows/     # 若使用 GitHub Actions 發布
```

### 7.2 必備文件清單（複製至新專案時）

| 複製來源（本專案） | 新專案放置 | 備註 |
| :--- | :--- | :--- |
| `internal_docs/公司專案標準規範_需求與開發協作.md` | `internal_docs/` | 公司標準，可全文複製 |
| `internal_docs/協作與治理規範.md` | `internal_docs/` | 專案名稱、Repo、角色名稱替換 |
| `internal_docs/docs_治理規範.md` | `internal_docs/` | 若對外僅首頁可保留；目錄名可沿用 |
| `internal_docs/程式編碼原則_共用規範.md` | `internal_docs/` | 共用；新專案另建「本專案程式代號」時可引用 |
| `internal_docs/新專案參考使用辦法_整合AI開發.md` | `internal_docs/` | 本文件，作新專案檢核與套用參考 |
| `00_Templates/Tpl_PRD_功能作業.md` | `00_Templates/` | PRD 範本 |
| `00_Templates/Tpl_權限矩陣表.md` | `00_Templates/` | 權限矩陣範本 |
| `00_Templates/Tpl_模組欄位名稱對照表.md` | `00_Templates/` | 欄位對照表範本 |
| `.cursorrules` |  Repo 根目錄 | 專案類型與 Repo 路徑可替換；**協作與治理、文件治理路徑**改為新專案之 `internal_docs/` |

可選：`internal_docs/提交變更步驟.md`、`internal_docs/更新對外文件與發布.md`、`internal_docs/如何要求文件優化.md`、`internal_docs/GitHub_檢查清單.md`（內容依新 Repo 調整）。

### 7.3 新專案啟動檢查清單

- [ ] 已建立上述目錄結構（至少 00_Templates、01_Discovery、02_Analysis、docs、internal_docs）
- [ ] 已複製公司專案標準規範、協作與治理規範、文件治理規範至 `internal_docs/`
- [ ] 已複製或建立 .cursorrules，並指向 `internal_docs/` 內之協作與治理、文件治理
- [ ] 已複製 PRD、權限矩陣、欄位對照表範本至 `00_Templates/`
- [ ] 協作與治理規範內之**專案名稱、Repo 網址、需求方／開發方稱謂**已替換為新專案
- [ ] 若為 e化 專案：已約定**模組代號**與**主程式代碼**產出方式（可先建一版「模組與程式代號架構」於 02_Analysis 或 internal_docs）
- [ ] 已約定：需求方授權 Repo 給開發人員、每週同步與客戶回饋通知方式
- [ ] 已約定：涉及需求／PRD／雛型／docs 的 PR 由需求方審核（建議 main 設 Branch protection）
- [ ] （可選）已建立 `docs/project_overview.md` 作為對外首頁，並設定 MkDocs 或 GitHub Pages
