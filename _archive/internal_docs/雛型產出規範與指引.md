# 雛型產出規範與指引（HTML+JS）

本文件規範**半成品雛型**（HTML+JS）之產出要求，使雛型具備**操作一致性、響應式畫面、欄位說明一致性、資安考量**，並可作為**操作說明文件**與**訓練／模擬操作網站**之來源。產出或審核雛型時請依本指引與 [模組與程式代碼編碼原則](模組與程式代碼編碼原則.md) 雛型命名一併依循。

---

## 1. 適用情境與目的

| 項目 | 說明 |
| :--- | :--- |
| **適用產物** | 以 HTML+JS 模擬畫面與操作流程，提供客戶確認之半成品雛型 |
| **目的** | 需求確認、操作一致性、響應式呈現、欄位說明可追溯、資安示範、**可轉為操作手冊與訓練模擬來源** |

---

## 1.1 版本保留與必備內容（分析階段）

為求**作業快速**，雛型會多次迭代時請遵守：

| 項目 | 規則 |
| :--- | :--- |
| **版本保留** | **分析過程中雛型一律只留 2 個版本**（最新版 + 前一版）；其餘舊版可刪除或歸檔至 99_Archives，避免堆積。 |
| **程式說明** | 每個雛型**一律**附**程式說明**（如：對應主程式代號、畫面用途、與權限矩陣之對應）；可置於雛型頁面頂部區塊或同目錄之說明檔。 |
| **操作引導** | 每個雛型**一律**附**操作引導**（如：操作步驟、按鈕說明、必填欄位、送出後預期行為）；可為頁面內「操作引導」區塊、註解或同目錄之短說明。 |
| **JS 與資料模擬互動** | 雛型須**加上 JS 與資料**，能**模擬互動**（如表單驗證、按鈕反應、列表篩選、步驟切換），而非僅靜態 HTML；假資料可內嵌於 `<script>` 或獨立 .js，以利客戶實際操作確認。 |

---

## 2. 操作一致性（Operational Consistency）

同一專案內之雛型，操作與互動方式應一致，以利客戶習慣與後續轉為說明文件、訓練站時結構統一。

### 2.1 導覽與版面

| 項目 | 建議 |
| :--- | :--- |
| **主選單／麵包屑** | 同一專案採用相同位置（如頂欄或側欄）、相同展開/收合邏輯；與程式代號架構（x00 目錄）對齊 |
| **返回／取消／送出** | 按鈕相對位置一致（如送出在右、取消在左）；同一動作之用語一致（全用「送出」或全用「確認」） |
| **表單流程** | 多步驟表單之「上一步／下一步」、進度指示方式一致 |

### 2.2 互動與回饋

| 項目 | 建議 |
| :--- | :--- |
| **成功／錯誤訊息** | 同一呈現方式（如頂部橫幅、modal、inline）；雛型可先用 `alert` 或固定區塊示意，但**位置與觸發邏輯**專案內一致 |
| **必填／驗證** | 必填標示方式一致（* 或「必填」）；驗證時機一致（送出時或離欄時），便於文件化為「操作步驟」 |
| **載入狀態** | 若有非同步模擬，載入中指示方式一致（spinner、遮罩、文字） |

### 2.3 與程式代號對齊

- 每個雛型對應**單一主程式代號**。**分析階段**工作檔可採 `YYYY-MM-DD_prototype_<主程式代號>.html` 以區分迭代；**對外發布至 MkDocs** 時依 §7.2 改為「程式代號+程式名稱」、不分版次。
- 同一主程式內之多畫面（如建檔內子步驟），建議以**同一 HTML 內區塊切換**或**明確註記對應子編碼**，利於與權限矩陣、操作手冊章節對應。

---

## 3. 響應式畫面（Responsive Layout）

雛型應在桌面與平板／手機寬度下皆可閱讀與操作，以利客戶於不同裝置確認，並作為未來訓練模擬與操作說明之截圖來源。

### 3.1 基本要求

| 項目 | 建議 |
| :--- | :--- |
| **viewport** | `<meta name="viewport" content="width=device-width, initial-scale=1">` 必備 |
| **斷點** | 至少考量：桌面（如 ≥1024px）、平板（768px～1023px）、手機（&lt;768px）；表單與按鈕可觸控區域建議 ≥44px |
| **字級與間距** | 使用相對單位（rem/em）或至少不固定 px 過小；行高與按鈕間距在縮小寬度時仍可讀可點 |

### 3.2 實作建議（雛型可簡化）

- 使用 **CSS Media Queries** 或單一欄位在窄版改為直向堆疊。
- 若雛型為靜態展示，可提供「桌面版／手機版」切換連結或註記，說明正式系統將為響應式。
- 未來轉為訓練模擬站時，同一套 HTML 可沿用相同 breakpoint，減少重做。

---

## 4. 欄位說明一致性（Field Description Consistency）

欄位標籤、placeholder、說明文字應與**模組欄位名稱中英文對照表**（及 PRD 四項必備之欄位對照表）一致，並利於擷取為操作說明與訓練教材。

### 4.1 單一來源對齊

| 項目 | 建議 |
| :--- | :--- |
| **欄位中文名稱** | 與「模組欄位名稱中英文對照表」之**中文**一致；若有簡稱，對照表應註明 |
| **欄位英文／代碼** | 表單欄位之 `name`、`id` 或 `data-field` 與對照表之**英文／縮寫**對齊，利於文件與程式對應 |
| **說明文字** | 欄位旁之說明（tooltip、小字說明）與對照表之**用途**欄或 PRD 情境說明一致，避免同一欄位在不同雛型寫法不同 |

### 4.2 可擷取為文件與訓練的結構（建議）

為使雛型可轉為**操作說明文件**與**訓練模擬操作網站**，建議在 HTML 中為「畫面／步驟／欄位」加上**結構化屬性**，供後續自動或半自動擷取：

| 屬性或結構 | 用途 | 範例 |
| :--- | :--- | :--- |
| **data-screen** 或 **data-step** | 標示畫面或操作步驟編號，對應手冊章節或訓練課程步驟 | `<section data-screen="CUS110-01">`、`<div data-step="2">` |
| **data-field-id** 或 **aria-label** | 欄位與對照表之對應；無障礙與文件擷取 | `<input data-field-id="CUS_APPLY_NO" aria-label="申請單號">` |
| **欄位說明區塊** | 同一區塊內集中「欄位代碼、中文名稱、說明」，方便轉成表格或手冊附錄 | `<dl class="field-desc"><dt>申請單號</dt><dd>…</dd></dl>` 或註解 |
| **註解區塊** | HTML 註解中註明「操作說明：…」「訓練重點：…」，後續可擷取為手冊或訓練腳本 | `<!-- 操作說明：請先選擇類別再填單號 -->` |

以上可依專案工具鏈簡化，但**欄位名稱與對照表一致**為必要；結構化屬性與註解為建議，以利未來產出操作手冊與訓練站。

## 5. 資安要求考量（Security Considerations）

雛型雖為半成品，仍應符合公司 e化 標準之**雛型設計資安建議**，並與正式系統資安要求對齊。詳見 [Tpl_GOV_公司e化標準治理](../00_Templates/Tpl_GOV_公司e化標準治理.md) 第二部分 §2.5。

### 5.1 雛型產出時必守

| 類別 | 要求 |
| :--- | :--- |
| **授權與權限** | 依權限矩陣表與情境，在雛型中**區分角色**示範（如標示「僅 XX 角色可見」、不同角色看到不同選單或按鈕）；避免僅產出「全部功能可見」單一版本 |
| **敏感資料** | 敏感欄位以**遮罩**（如 `***`）或**假資料**示範；**不使用真實**客戶、個資或業務資料；若有範例資料，頁面或註解註記「僅供示意，正式環境不得使用真實個資」 |
| **登入與登出** | 若含登入／操作流程，保留**登出**按鈕或連結示意；可註記「逾時登出與 Session 管理由正式系統實作」 |
| **傳輸與環境** | 於說明或註解註明「正式環境須以 **HTTPS** 提供」；表單送出行為註記「實際送出須經後端驗證與加密」 |

### 5.2 與正式開發對齊

- 雛型中之權限區分、遮罩與假資料、登出示意，應與 PRD 及「協作與治理」§2.5 開發要求一致，供開發團隊實作時延續。

## 6. 可轉為操作說明與訓練模擬來源（Documentation & Training Source）

雛型設計時預留「可轉為文件與訓練站」之結構，可減少重複產出與不一致。

### 6.1 操作說明文件

| 目標 | 建議做法 |
| :--- | :--- |
| **章節對應** | 每個雛型對應主程式代號；雛型內 `data-screen`／`data-step` 可對應手冊之「作業／步驟」 |
| **擷取欄位說明** | 欄位對照表＋雛型內 `data-field-id`／label，可產出「畫面欄位說明」附錄或表格 |
| **擷取操作步驟** | 註解區塊「操作說明：…」或按鈕/連結之 aria-label、data-step，可彙整為操作步驟清單 |
| **截圖來源** | 響應式雛型在固定斷點下截圖，即可用於手冊之「畫面說明」；建議同一畫面保留桌面/手機版截圖規則一致 |

### 6.2 訓練／模擬操作網站

| 目標 | 建議做法 |
| :--- | :--- |
| **同一 HTML 來源** | 雛型 HTML+JS 可作為訓練站之**同一來源**；僅需加上「訓練模式」標示、提示或腳本（如高亮步驟、限制操作順序），不必重做畫面 |
| **步驟與情境** | 利用 `data-step`、`data-screen` 與權限情境，可產出「角色 A 操作步驟」「角色 B 操作步驟」之訓練腳本或引導 |
| **假資料與環境** | 雛型已使用假資料與遮罩，訓練站可沿用同一假資料集，並註明「訓練環境」避免與正式混淆 |

### 6.3 產出時自問

- 若明天要從此雛型**自動或半自動產出一份操作手冊**，目前 HTML 結構與註解是否足以擷取「畫面名稱、步驟、欄位說明」？
- 若明天要將此雛型**改為訓練模擬站**，是否僅需加一層「訓練模式」邏輯，而不必改動主要版面與流程？

## 7. 雛型檔案與目錄（提醒）

| 項目 | 依循 |
| :--- | :--- |
| **分析階段檔名** | 工作檔可採 `YYYY-MM-DD_prototype_<主程式代號>.html` 以區分迭代；**僅保留最新 2 版**，其餘刪除或歸檔。 |
| **對外發布檔名** | 生成對外文件時，**一律用「程式代號+程式名稱」命名**，**檔名不分版次**；版次於 HTML 內容或標題呈現。見 §7.2。 |
| **放置** | 分析期可置 `02_Analysis/` 或專案約定目錄；對外發布時將**最新版**更新至 `docs/prototype/` 並以「程式代號+程式名稱」檔名覆蓋或單一檔；對內草稿可置 `internal_docs/prototype/`。**目錄與路徑請使用英文**（如 `prototype/`），避免中文路徑造成轉檔或部署亂碼。 |
| **共用的 CSS/JS** | 為達操作一致性，建議專案內有**共用片段**（如共用的按鈕樣式、表單驗證、導覽列）以內聯或共用檔引用，避免每個雛型重寫一份。 |

### 7.1 命名規則：避免轉檔亂碼

雛型將列於**提供給客戶的 MkDocs 網站**之「功能與說明」導覽中；為避免建置、部署或不同環境因編碼差異造成**路徑或檔名亂碼**，**檔名僅使用英文、數字、連字號、底線**，禁止中文。顯示用程式名稱可用中文（表格、標題）。

### 7.2 對外發布至 MkDocs：程式代號+程式名稱、不分版次

生成**對外文件**時，將雛型**最新版**更新到 MkDocs（如複製至 `docs/prototype/` 並更新 `docs/features.md` 連結）。對外雛型檔名規則：

| 項目 | 規則 | 範例 |
| :--- | :--- | :--- |
| **檔名格式** | **一律用「程式代號 + 程式名稱」命名**，**檔名不分版次**（不寫日期或 v1、v2） | ✅ `CUS110_ExportCase.html`、`CUS100_ExportMenu.html` |
| **程式名稱** | 程式名稱為中文時，檔名請用**英文縮寫或羅馬拼音**，避免中文造成亂碼 | ✅ `CUS110_ExportCase`（出口案件建檔） ❌ `CUS110_出口案件建檔.html` |
| **版次呈現** | 版次**不寫在檔名**，改於 **HTML 內容或標題**呈現（如 `<title>`、頁首「v2」「更新日期 2025-02-20」） | 客戶開啟雛型後從頁面即可見版次或更新日 |
| **連結** | MkDocs 功能與說明頁之雛型連結，指向上述單一檔名（無版次） | `[雛型檢視](prototype/CUS110_ExportCase.html)` |

總結：**對外雛型一程式一檔、檔名=程式代號_程式名稱英文；版次僅在 HTML 內標題或內容顯示。**

## 8. 雛型產出檢查清單

產出或審核雛型（HTML+JS）時，建議依下列項目勾選：

### 8.1 操作一致性

- [ ] 導覽／主選單位置與邏輯與同專案其他雛型一致
- [ ] 按鈕（返回、取消、送出）位置與用語一致
- [ ] 成功／錯誤訊息呈現方式一致
- [ ] 雛型檔名與主程式代號對齊，且與權限矩陣一致
- [ ] **命名避免亂碼**：雛型檔名與路徑僅使用英文、數字、`-`、`_`，無中文；MkDocs 對外頁面檔名建議英文（如 features.md）

### 8.2 響應式

- [ ] 已設定 viewport meta
- [ ] 至少一種窄版（如手機寬度）可讀可操作，或已註記「正式系統將為響應式」
- [ ] 可觸控區域與字級在窄版下仍可用

### 8.3 欄位說明一致性

- [ ] 欄位標籤／placeholder 與「模組欄位名稱中英文對照表」一致
- [ ] 欄位 `name`／`id`／`data-field-id` 與對照表英文／縮寫對齊（或已註記對應）
- [ ] 若有欄位說明文字，與對照表用途或 PRD 一致

### 8.4 資安

- [ ] 已依權限矩陣區分角色示範（或註記角色）
- [ ] 敏感欄位已遮罩或使用假資料，無真實個資／業務資料
- [ ] 若有登入流程，含登出示意；已註記 HTTPS／後端驗證（若適用）

### 8.5 必備：程式說明與操作引導、JS 模擬互動

- [ ] **程式說明**：雛型附程式說明（主程式代號、畫面用途、與權限對應）；可為頁面區塊或同目錄說明
- [ ] **操作引導**：雛型附操作引導（步驟、按鈕說明、必填、送出後預期）；可為頁面區塊、註解或短說明
- [ ] **JS 與資料**：已用 JS 與假資料模擬互動（表單驗證、按鈕反應、列表或步驟切換等），非僅靜態 HTML
- [ ] **版本**：分析階段僅保留最新 2 版；對外發布時以「程式代號+程式名稱」單一檔名、版次於 HTML 內呈現

### 8.6 可轉為說明與訓練

- [ ] 已加 `data-screen`／`data-step` 或等同結構，可對應手冊章節／訓練步驟（或已規劃後補上）
- [ ] 欄位可對應至對照表，利於擷取為欄位說明附錄
- [ ] 必要處有「操作說明」或「訓練重點」註解，利於產出操作手冊或訓練腳本
