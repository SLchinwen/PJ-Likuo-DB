# e化專案進行與工程分工評估

本文件整理目前**e化專案進行狀況**與**與工程方（矽聯）之分工方法**，並評估合理性、優缺點與改進建議，供內部檢討與優化參考。

---

## 1. 現行作法摘要

### 1.1 專案進行狀況（報關管理模組）

| 項目 | 現況 |
| :--- | :--- |
| 模組範圍 | 報關管理，六類主作業（出口、進口、轉運、機場、SP、船用品） |
| 需求產出 | 僅**出口**完成訪談與作業說明／欄位定義；其餘五類僅完成表單畫面整理，尚未訪談 |
| 主程式代號 | 六類均已編列（目錄、建檔、查詢含增修、歷史歸檔），權限表用 |
| PRD／雛型 | 出口已有雛型與 PRD、客戶回饋進版；其餘待訪談後產出 |
| 標準產出要求 | 已訂：模組與主程式代碼、權限矩陣表、情境說明（三項必備），作為框架／TABLE／API 命名基礎 |

### 1.2 與工程方之分工方法（協作協議）

| 項目 | 作法 |
| :--- | :--- |
| 需求方 | 分享**本 GitHub 專案**（需求、PRD、雛型、文件）予工程方參考 |
| 工程方 | 依需求方資料，在**自有之工程／開發專案**內建立專案管理與實作；**不使用同一份 GitHub**，避免互相衝突 |
| 迭代與變更 | 需求變更或迭代時，由**工程方在其專案**內進行變更與差異分析；需求方 PRD／雛型更新以本 repo 為準，供工程方對照與同步 |
| 文件版本 | **文件版本與發布由需求方自行管理**；對外與客戶交付以需求方 repo 為依據 |
| PRD 產出 | 各功能作業 PRD 須先建立模組與主程式代碼，並提供**程式×角色權限矩陣表**與**對應情境說明**，作為框架、TABLE、API 命名與權限規劃基礎 |

## 2. 合理性評估

### 2.1 整體判斷：**合理，且符合常見外包／協作情境**

| 面向 | 評估 |
| :--- | :--- |
| **權責邊界** | 需求方握有需求與文件主權，工程方專注實作與專案管理，邊界清楚。 |
| **版本與衝突** | 不同 repo 可避免 commit 衝突、權限與審核流程混雜；需求方單一來源（single source of truth）利於對外口徑一致。 |
| **交付依據** | PRD 三項必備（代碼、權限矩陣、情境）提供工程方明確的命名與權限設計依據，降低溝通成本。 |
| **擴充性** | 主程式代號與共用規範已訂，後續不同專案／模組可套用，有利長期 e化 擴充。 |

### 2.2 需留意之前提

- **同步機制**：需求方更新 PRD／雛型後，須有約定方式（如通知、定期同步、版本號）讓工程方即時對照，避免兩邊版本分叉。
- **差異分析責任**：迭代時「由工程方在其專案內進行變更與差異分析」須約定產出形式（如差異清單、影響範圍表），以便需求方驗收與追蹤。
- **單一可開發範圍**：目前僅出口具完整 PRD／欄位定義，其餘五類尚未訪談；若工程方要全面鋪開，須等需求方補齊訪談與 PRD，或先鎖定出口為第一階段範圍。

## 3. 優點

| 優點 | 說明 |
| :--- | :--- |
| **避免版本與權限衝突** | 不同 repo 下，需求方不必處理程式碼 merge、branch 保護等，工程方可依自身流程建置、測試、部署。 |
| **需求方主導文件與發布** | 文件版本與對外發布由需求方管理，客戶與內部只看一份來源，利於品牌與合約責任歸屬。 |
| **需求產出有結構可依** | 模組與主程式代碼、權限矩陣、情境說明形成固定產出結構，工程方易對齊框架、TABLE、API 命名與權限實作。 |
| **迭代責任清楚** | 變更與差異分析在工程方專案內完成，需求方以「更新本 repo 的 PRD／雛型」為輸入，責任歸屬明確。 |
| **可複用至其他專案** | 程式編碼原則、共用規範、標準分析產出要求可套用至不同 e化 專案或模組。 |

## 4. 缺點與風險

| 缺點／風險 | 說明 | 緩解方向 |
| :--- | :--- | :--- |
| **雙邊同步依賴人為** | 需求方更新後若未主動通知或標註版本，工程方可能沿用舊版 PRD，造成實作與需求不一致。 | 約定「PRD／雛型更新即通知」或版本號／變更清單；必要時工程方定期拉取或檢視需求方 repo。 |
| **差異分析產出未定型** | 「工程方在其專案內進行變更與差異分析」若未約定產出格式，需求方難以驗收與追蹤影響範圍。 | 約定差異分析產出（如變更清單、影響程式／TABLE／API 對照表），並可納入交付檢查清單。 |
| **需求方為單點** | 文件與發布集中需求方，若人力或交接不順，可能影響工程方取得最新規格。 | 文件結構化、範本化（已完成）；重要里程碑可打 Tag／版次，便於對應。 |
| **僅出口可開發** | 其餘五類尚未訪談與 PRD，工程方若期待全模組並行開發，會卡在需求未齊。 | 明確約定第一階段範圍（建議以出口為主），其餘待訪談與 PRD 完成後再排程。 |
| **無共同工單／議題追蹤** | 若兩邊各自用不同工具（如需求方用本 repo issue、工程方用自建看板），需求變更與 bug 可能分散。 | 可約定：需求變更與驗收條件以需求方 repo issue 或 PR 描述為準，工程方對應自家工單時引用需求方編號或連結。 |

## 5. 建議

### 5.1 短期（維持現行分工下）

| 建議 | 作法 |
| :--- | :--- |
| **約定「需求更新」通知方式** | 需求方更新 PRD、雛型或程式代號架構後，以約定管道（如 Email、群組、一則 issue）通知工程方，並註明版次或變更摘要。 |
| **鎖定第一階段開發範圍** | 正式約定以**出口作業**為第一階段開發與上線範圍；進口～船用品待訪談與 PRD 完成後再納入排程，減少範圍不明。 |
| **差異分析產出格式** | 與工程方約定：迭代或變更時，工程方提供簡要**變更／差異清單**（如影響之程式、TABLE、API 或畫面），供需求方驗收與留檔。 |
| **交付定義寫入協作規範** | 在協作與治理規範中明列「一輪交付」的定義（例如：需求方已 Approve、通過驗收清單、文件已更新），減少認知落差。 |

### 5.2 中期（優化協作）

| 建議 | 作法 |
| :--- | :--- |
| **版本對照表** | 需求方在對外或內部文件中維持「PRD／雛型版次與發布日」表；工程方於其專案標註「本版對應需求方 repo 之版次／commit／Tag」，便於對回。 |
| **權限矩陣與情境的驗收** | 第一階段（出口）上線前，將「權限矩陣表」與「情境說明」納入驗收項目，確認角色與程式權限、情境流程與實作一致。 |
| **簡易變更清單範本** | 需求方提供「需求變更清單」範本（變更項目、影響程式代號、預期行為），供每次迭代時填寫並交工程方，利於差異分析與估價。 |

### 5.3 長期（視需要）

| 建議 | 作法 |
| :--- | :--- |
| **其他模組訪談與 PRD 節奏** | 進口、轉運、機場、SP、船用品排定訪談與 PRD 產出時程，每完成一類即補齊三項必備產出，再交給工程方排程，避免一次大爆炸。 |
| **共同檢查點** | 每季或每階段做一次「需求方 repo 與工程方專案對照」檢查（範圍、版次、已知差異），及早發現分叉。 |
| **知識留存** | 關鍵決策（如主程式代號取捨、權限矩陣定案）寫入本 repo 之 PRD 或 internal_docs，降低人員異動後的銜接成本。 |

---

## 6. 結論

- **現行 e化 專案進行與工程分工方式整體合理**：需求方掌握需求與文件、工程方自有專案實作與迭代、不同 repo 避免衝突、文件版本由需求方管理，符合常見外包／協作模式。
- **主要優點**：權責清楚、避免版本衝突、需求產出有結構（代碼、權限矩陣、情境）、可複用至其他專案。
- **主要風險**：雙邊同步依賴人為、差異分析產出未定型、目前僅出口可完整開發；可透過「更新通知、差異清單、鎖定第一階段範圍、交付定義與驗收清單」等短期建議緩解。
- 建議優先落實**短期建議**（通知方式、第一階段範圍、差異分析產出、交付定義），再視合作情況導入中期與長期優化。
