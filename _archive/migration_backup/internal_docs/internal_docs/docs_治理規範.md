# docs 文件治理規範

本規範區分**對外（使用者）**與**內部（開發團隊）**文件，使產出與歸檔時能正確放置與撰寫。產生或新增文件時請依下列規則判斷歸屬。

---

## 1. 文件分類與目錄

| 對象 | 目錄 | 導覽區塊 | 用途 |
| :--- | :--- | :--- | :--- |
| **使用者／客戶** | `docs/` 根目錄 | 對外網站首頁（= 專案總攬） | 系統說明、作業模式、程式清單與雛型連結、上線後操作手冊與 SOP |
| **開發團隊** | `internal_docs/` | 不透過 MkDocs 發布 | 程式編碼原則、協作與分支、專案管理、提交步驟、文件撰寫指引、設計與 PRD 索引 |

**判斷原則**：內容是否會給**客戶或 end-user** 看？若是 → 對外（`docs/`）；若為專案管理、設計、開發流程、內部規範 → 內部（`internal_docs/`）。內部文件**不納入 MkDocs 建置**，不對外發布、不被搜尋。

---

## 2. 對外文件（使用者）規則

**放置位置**：`docs/index.md`（對外網站首頁即此檔），以及後續之操作手冊、SOP（可另建 `docs/manuals/` 等）。

**禁止出現**：

- 設計文件或分析文件之**路徑**（如 `02_Analysis/…`、`internal_docs/…`）
- 內部**狀態**（如 Draft、待規劃、開發中）
- 模組「規劃範圍」「待規劃」等**規劃狀態**描述
- 版本控制、分支、PR、審核流程等**開發流程**細節
- 檔案命名規則、儲存位置等**實作細節**

**可出現**：

- 系統與專案**目標**、**作業模式**、**程式清單**（表格：程式代號、程式名稱、用途、雛型）
- **雛型**：僅以**連結**呈現（不寫實體路徑）
- 文件**版次**與**發布日期**（表格式即可）
- 完成後將提供**操作手冊、SOP、教育訓練**等說明

**雛型連結**：對外只寫如 `[雛型檢視](prototype/YYYY-MM-DD_prototype_CUS110.html)`，不寫 `docs/prototype/...` 或檔名規則。

---

## 3. 內部文件（開發團隊）規則

**放置位置**：`internal_docs/` 下所有 Markdown 與子目錄（如未來之 `internal_docs/project/` 專案管理、紀錄）。

**可出現**：

- 設計文件、PRD、雛型規範之**路徑**與**索引**
- 文件或專案**狀態**（Draft、已發布、待規劃）
- **程式編碼原則**、雛型 HTML 命名與發布路徑
- **協作與治理**、分支、PR、提交步驟
- **專案管理**文件與紀錄（會議、決策、變更歷程）
- 如何要求文件優化、撰寫範本等**產出指引**

**歸檔**：專案管理或開發過程產生的紀錄，應放在 `internal_docs/` 或約定子目錄，並在 `internal_docs/index.md`（開發團隊首頁）列出，避免與對外首頁混在一起。

---

## 4. 對外網站（MkDocs）與內部文件

- **對外網站**：MkDocs 僅發布 **首頁**，且首頁直接對應 `docs/index.md`（專案總攬）。無其他導覽項目，內部文件**不納入建置**，故不對外、不搜尋。
- **內部文件**：僅存於 repo 之 `internal_docs/`，供團隊在 Cursor／Git 中查閱；不透過 MkDocs 發布。

新增文件時：對外 → 更新 `docs/index.md` 或於 `docs/` 新增對外頁面並於 mkdocs nav 加入；內部 → 置於 `internal_docs/`，無需改 mkdocs。

---

## 5. 產出時自檢

| 情境 | 應放置 | 內容注意 |
| :--- | :--- | :--- |
| 給客戶看的總覽、程式清單、雛型入口 | `docs/`（index 即首頁） | 無路徑、無狀態、雛型僅連結 |
| 操作手冊、SOP、教育訓練教材 | `docs/`（或 docs 下對外子目錄） | 同對外規則 |
| 程式代號、編碼原則、雛型命名規範 | `internal_docs/` | 可有路徑與範例 |
| **雛型設計／生成** | `02_Analysis/` 或 `docs/prototype/` | 依 **§7** 檢視：**一般要求**（操作一致性、響應式畫面、欄位說明與對照表一致、可轉操作說明與訓練模擬來源）；**資安建議**（權限區分、敏感欄位遮罩、登出／逾時、HTTPS 與後端驗證註記） |
| 協作、分支、PR、提交步驟 | `internal_docs/` | — |
| 專案管理、會議紀錄、變更紀錄 | `internal_docs/`（或子目錄） | — |
| 設計文件、PRD 本體 | 維持 `02_Analysis/`，在 internal_docs 做**索引** | 對外文件不直接連結 |
| **功能作業 PRD**（含權限矩陣、情境說明、欄位對照表） | `02_Analysis/` | 須符合 **§6 標準分析產出要求**（模組與主程式代碼、權限矩陣表、情境說明、模組欄位名稱中英文對照表四項必備） |

依上述規則產生或搬移文件，即可維持「使用者看到的簡潔對外說明」與「開發團隊的完整歸檔」分離。

## 6. 標準分析產出要求（e化專案）

本專案為 e化專案，**各功能作業之 PRD 與分析產出**須符合下列**標準分析產出要求**，作為工程方**框架設計、TABLE 命名、API 命名與權限設計**之依據。詳見 [協作與治理規範](協作與治理規範.md) §2.4。

### 6.1 產出項目（四項必備）

| 項目 | 說明 | 範本／參考 |
| :--- | :--- | :--- |
| **模組與主程式代碼** | 該功能作業所屬模組及**完整**主程式代號清單（權限表用）；須先建立完成再撰寫 PRD。子程式、批次、API／TABLE 命名依 [程式編碼原則｜共用規範](程式編碼原則_共用規範.md) §4～§5。 | [程式編碼原則｜共用規範](程式編碼原則_共用規範.md)、各模組之程式代號架構文件 |
| **程式 × 使用者角色權限矩陣表** | 依情境產出：主程式代號 × 使用者角色之權限矩陣（可執行／可查詢／不可見等）。 | [Tpl_權限矩陣表](../00_Templates/Tpl_權限矩陣表.md) |
| **對應情境說明** | 各使用情境之操作者角色、可存取程式、流程與驗收說明。 | [Tpl_PRD_功能作業](../00_Templates/Tpl_PRD_功能作業.md) 內「情境說明」章節 |
| **模組欄位名稱中英文對照表** | 該模組**所有客戶作業表單與報表**之欄位：**中文、英文、縮寫、用途**表列；與客戶對齊表單與報表名稱，新欄位時提醒**新增／對應／修改／刪除**，避免需求失真。 | [Tpl_模組欄位名稱對照表](../00_Templates/Tpl_模組欄位名稱對照表.md) |

### 6.2 產出順序與用途

1. **先建立**模組與主程式代碼（含目錄、建檔、查詢含增修、歷史歸檔等），再撰寫該功能作業 PRD。
2. **依情境**產出權限矩陣表與情境說明，與主程式代號對齊。
3. **同時建立或更新**該模組之**欄位名稱中英文對照表**，涵蓋該模組所有客戶作業表單之欄位（中文、英文、縮寫、用途）；專案可能分開時段建置，此表可讓客戶與我方持續對齊表單與報表名稱，若有新欄位出現可提醒新增／對應／修改／刪除，避免後續需求失真。
4. 上述產出供工程方用於：**框架**、**資料表（TABLE）**、**API** 之命名與權限規劃，PRD 須一併提供或引用。

### 6.3 放置與歸檔

- 功能作業 PRD 本體：`02_Analysis/`（或專案約定之 PRD 目錄）。
- 權限矩陣表：可獨立成檔（如 `02_Analysis/YYYY-MM-DD_模組名_權限矩陣.md`）或併入 PRD 章節。
- **欄位名稱對照表**：建議獨立成檔（如 `02_Analysis/YYYY-MM-DD_模組名_欄位名稱對照.md`），該模組僅維護一份，涵蓋所有作業表單與報表欄位；有新表單或新欄位時更新同一檔並註記異動。
- 程式代號架構：`02_Analysis/` 或 `internal_docs/`，並在 internal_docs 索引中列出。

### 6.4 產出時自檢（功能作業 PRD）

產出或審核功能作業 PRD 時，請確認：已附或已引用**模組與主程式代碼**、已附**程式 × 角色權限矩陣表**、已附**對應情境說明**、已附或已更新**模組欄位名稱中英文對照表**；四者齊備後再提供工程方，作為框架與 TABLE／API 命名規劃基礎，並與客戶對齊表單與報表名稱。

### 6.5 需求與情境流程審核建立歸檔時之拆分提醒

在**需求與情境流程審核、建立、歸檔**時，建議採用 [PRD 拆分與交付單元規則](standards/PRD_拆分與交付單元規則.md) 之**拆分最小邏輯**：

- **一份 PRD 僅涵蓋一個主程式＋其子程式**，不寫整個專案
- 採用此邏輯可有助於**分給工程團隊平行開發**，不同主程式可分配不同工程師同時進行
- 分開交付、需求變更影響範圍可控

產出或歸檔需求、情境說明、流程圖時，可依主程式邊界劃分，便於後續分開產出 PRD 與交付開發包。

### 6.6 產出交付開發包時之檢查（子程式清單、欄位對照表）

產出或組裝**交付開發包**時，須依下列規範完成檢查：

| 檢查項 | 說明 | 規範 |
| :--- | :--- | :--- |
| **子程式清單必備** | 有 UI 之主程式交付包須含 `01_規格/子程式清單.md` | [子程式清單與實作建議｜治理規範](standards/子程式清單與實作建議_治理規範.md) |
| **欄位名稱對照表必備** | 有 UI 之主程式交付包須含 `01_規格/欄位名稱對照表.md` | [需求釐清與交付包｜治理規範](standards/需求釐清與交付包_治理規範.md) §1 |
| **自動產出** | 若尚未產出，AI 應依雛型與需求**自動產出**並納入交付包 | 同上 |
| **主檔匯出入** | 單檔主檔類須於需求註明整批匯出與匯入 | [需求釐清與交付包｜治理規範](standards/需求釐清與交付包_治理規範.md) §2 |
| **資料來源說明** | 跨程式資料來源須於 PRD 含「資料來源與關聯」章節 | [需求釐清與交付包｜治理規範](standards/需求釐清與交付包_治理規範.md) §3 |

AI 實作時依子程式清單與 [子程式實作建議｜標準](standards/子程式實作建議_標準.md) 展開工作包。TABLE 與 API 設計見 [TABLE 與 API 設計｜AI 產出指引](standards/TABLE與API設計_AI產出指引.md)。工程師開工指引見 [工程師開工指引｜治理規範](standards/工程師開工指引_治理規範.md)。

## 7. 雛型設計與產出（含資安建議）

雛型之**命名與放置**依 [程式編碼原則｜共用規範](程式編碼原則_共用規範.md)（雛型 HTML 檔名建議：`YYYY-MM-DD_prototype_<程式代號>.html`）；對外呈現依本規範 §2（雛型僅以連結呈現，不寫實體路徑）。

### 7.1 雛型產出之一般要求

生成雛型時須符合下列要求，以利與客戶對齊、開發對接，並作為後續**操作說明文件**與**訓練／模擬操作網站**之來源。

| 要求項目 | 說明 |
| :--- | :--- |
| **操作一致性** | 同一專案內雛型之**操作方式**須一致：按鈕位置與樣式、表單送出／返回／取消行為、選單展開與收合、錯誤或成功提示方式等，跨程式代號保持相同慣例，以降低使用者與開發方認知落差。 |
| **響應式畫面** | 雛型須具**響應式**考量：在桌面、平板、手機等不同螢幕寬度下，版面可讀、表單可操作；若專案以桌面為主，至少約定一組最小寬度並在雛型中示範，以利未來正式系統與操作手冊截圖一致。 |
| **欄位說明之一致性** | 同一**欄位**（中文名稱、用途、選項）在雛型、PRD、**模組欄位名稱對照表**中須一致；欄位說明、placeholder、必填標示、單位等與 [模組欄位名稱中英文對照表](../00_Templates/Tpl_模組欄位名稱對照表.md) 及該模組實際對照表對齊，避免同一欄位在不同畫面或文件出現不同說法。 |
| **無障礙（a11y）可及性** | 表單控制項（`select`、`input`、`button` 等）須具備**可及性名稱**，以通過瀏覽器開發者工具（如 Microsoft Edge Tools）之可及性檢查；`select` 元素若無對應 `<label for="id">` 關聯，應加上 `aria-label` 或 `title` 屬性，說明該欄位用途，俾利螢幕閱讀器與輔助技術正確識別。 |
| **操作按鈕置於表列上方** | 查詢列表之「新增」「搜尋」等主要操作按鈕應置於**表列上方**（與搜尋列同排或緊鄰），避免查詢多筆時使用者需捲動至列表底部始能操作，讓操作盡可能簡單。 |
| **資安要求考量** | 須符合本規範 **§7.2 雛型設計時之資安建議規範**（權限區分示範、敏感欄位遮罩、登出／逾時示意、HTTPS 與後端驗證註記），並對齊 [協作與治理規範 §2.5](協作與治理規範.md) 專案整體資安要求。 |
| **可轉為操作說明與訓練模擬來源** | 雛型產出時宜預留**後續再利用**：結構與文案足以作為**操作說明文件**（如步驟說明、畫面截圖對應）之依據；並可作為**訓練用模擬操作網站**或教材來源，故畫面標題、區塊說明、按鈕名稱等宜完整、語意清楚，避免僅有代號或暫存文字。 |
| **程式操作應用說明產出** | 產出雛型時須**同步產出**「程式操作步驟說明」文件（Markdown）：依雛型版面整理區塊／欄位說明、操作步驟、雛型連結；放置於 `docs/prototype/`；雛型頁面建議提供「📖 操作說明」按鈕連結至該文件。 |

需求方或產出雛型者於設計與生成時依上表檢視，可提升雛型之可用性與後續文件化、訓練模擬之效益。

### 7.2 雛型設計時之資安建議規範

| 對應類別 | 雛型設計／生成時建議 |
| :--- | :--- |
| **授權與權限** | 依 **權限矩陣表** 與情境，在雛型中**區分角色**示範：標示或註記「此按鈕／連結僅 XX 角色可見」、不同角色看到之選單或功能區塊差異；避免僅產出單一「全部功能可見」畫面，以利與開發階段之最小權限對齊。 |
| **敏感資料** | 雛型中之**敏感欄位**（個資、報關單號、客戶資料等）以**遮罩**（如 `***`、`○○○`）或**假資料**示範，**不使用真實**客戶或報關資料；若為範例資料，註記「僅供示意，正式環境不得使用真實個資」。 |
| **登入與登出** | 雛型若含登入／操作流程，建議保留**登出**按鈕或連結之示意，並可註記「逾時登出與 Session 管理由正式系統實作」；與整體身分認證與登入資安要求對齊。 |
| **傳輸與環境** | 若雛型為靜態 HTML，可於說明或註記中註明「正式環境須以 **HTTPS** 提供」；表單或送出行為可註記「實際送出須經後端驗證與加密」，提醒實作階段不依前端為唯一防線。 |
| **對齊整體要求** | 雛型產出時一併參照 [協作與治理規範 §2.5](協作與治理規範.md) 之資安基本要求；若有客戶或公司額外資安／個資規定，於 PRD 或雛型說明中註記，供開發方實作時納入。 |

需求方或產出雛型者於**設計與生成雛型**時依上表檢視，可與專案整體資安要求一致，並減少開發階段認知落差。
