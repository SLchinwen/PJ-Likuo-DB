# AI 治理與開發銜接檢視與建議

本文件檢視現行**公司規範範本**與**文件生成邏輯**，是否符合**AI 治理需求**以及能否**銜接給開發團隊進行 AI 開發**使用，並提出具體建議。

---

## 1. 檢視結論摘要

| 面向 | 符合度 | 說明 |
| :--- | :--- | :--- |
| **AI 治理（需求方／顧問端）** | 部分符合 | 有範本、目錄、YAML、OPS、工程分工；.cursorrules 未引用 internal_docs 與程式編碼原則；AI 產出審核未明文寫入 .cursorrules |
| **銜接開發團隊** | 有缺口 | 缺「開發團隊 AI 開發入口」與 handoff 格式；.cursorrules 未指引 AI 讀取程式編碼原則；開發方 Repo 的必讀清單未標準化 |
| **文件生成邏輯** | 已具備 | 命名、歸類、四項必備、對內對外分離、檢查清單均已有定義，可作為 AI 產出依據 |

## 2. 符合 AI 治理需求之項目

### 2.1 已具備且可支援 AI 產出

| 項目 | 位置 | AI 治理用途 |
| :--- | :--- | :--- |
| **目錄與工作流** | .cursorrules、GOV_e化專案治理總則 | AI 可依 01_Discovery、02_Analysis、03_Solution、internal_docs 判斷產出放置位置 |
| **YAML + 版本紀錄** | 治理總則、範本 | AI 產出文件時可自動補檔頭與版本紀錄 |
| **檔案命名** | .cursorrules、治理總則 | `YYYYMMDD_類型_關鍵字.md` 可機械執行 |
| **OPS Runbooks** | OPS_Inbox_AutoSorter、OPS_Publish_To_Web | AI 可依步驟執行歸檔與發布，治理總則 §4 已註明「供人員或 **AI** 依步驟執行」 |
| **工程分工** | 治理總則 §5 | 明確定義 AI 助理責任與產物，避免越權 |
| **四項必備產出** | Tpl_GOV、新專案參考辦法 | PRD 須含模組與主程式代碼、權限矩陣、情境說明、欄位對照表，AI 產出 PRD 時可依自檢表勾選 |
| **對內對外分離** | Tpl_GOV、internal_docs | 判斷原則明確（給客戶看→docs；否→internal_docs），利於 AI 判斷 |
| **模組與程式代碼編碼原則** | internal_docs/模組與程式代碼編碼原則.md | 格式約束、禁止事項、檢查清單、TABLE/API 對齊，可供程式文件生成時依循 |
| **整合 AI 開發檢核** | _Inbox/新專案參考使用辦法_整合AI開發 | 已檢核單一需求來源、範本、.cursorrules、PRD 四項、資安、AI 產出審核歸屬、版本紀錄 |

### 2.2 已對齊「AI 產出須經審核」

- 新專案參考辦法 §6：**AI 產出之審核歸屬** ✅，涉及 01_Discovery、02_Analysis、docs、雛型之 PR 建議由需求方審核。
- 同文件**建議補強**：於 .cursorrules 或協作規範中明寫「AI 輔助產出之 PRD、雛型、文件須經需求方（或專案負責人）審核後再合併至 main」。

## 3. 缺口與風險（AI 治理與開發銜接）

### 3.1 .cursorrules 未引用 internal_docs 與程式編碼原則

| 缺口 | 影響 |
| :--- | :--- |
| .cursorrules 僅列 01_Discovery、02_Analysis、03_Solution、00_Templates，**未列 internal_docs** | AI 產出對內文件時，可能不知放置 internal_docs 或不知有內部規範可依循 |
| .cursorrules **未引用**「模組與程式代碼編碼原則」 | 產出 PRD、雛型檔名、程式清單、API/TABLE 命名時，AI 可能未讀取該原則，導致與規範不一致 |
| .cursorrules **未寫**「對內 vs 對外」判斷 | 產出文件時可能誤放或誤寫（如對外文件寫入內部路徑） |

### 3.2 缺「開發團隊 AI 開發」之明確入口與 handoff

| 缺口 | 影響 |
| :--- | :--- |
| 無**開發團隊 AI 開發指引** | 開發方用 Cursor/AI 時，不知應先讀 internal_docs 哪些文件、PRD 四項必備如何對應到程式/API 命名 |
| 無**handoff 格式** | 需求方交付 PRD＋雛型＋權限矩陣時，未標準化「交付清單」或「AI 必讀清單」，開發團隊 AI 易漏讀 |
| 程式編碼原則雖有檢查清單 | 但 .cursorrules 未指引「產出程式相關文件時請參照 internal_docs/模組與程式代碼編碼原則.md」，開發端 AI 可能未載入該檔 |

### 3.3 AI 產出審核未明文寫入本倉庫

| 缺口 | 影響 |
| :--- | :--- |
| 「AI 輔助產出須經需求方審核後再合併」僅在新專案辦法中建議補強，**本倉庫 .cursorrules 尚未補上** | 新成員或 AI 在本 Repo 操作時，可能未見審核要求，直接合併 |

### 3.4 新專案參考辦法與整合 AI 開發文件尚未歸檔

| 缺口 | 影響 |
| :--- | :--- |
| 「新專案參考使用辦法_整合AI開發」、相關規範仍在 **_Inbox** | 尚未放入 internal_docs，開發團隊或新專案複製時可能遺漏「整合 AI 開發」檢核與套用步驟 |

## 4. 建議（依優先級）

### 4.1 高優先：.cursorrules 補強（AI 治理）

1. **新增「對內文件」與 internal_docs 引用**  
   - 在 .cursorrules 的 Folder Structure 或獨立一節註明：**對內文件**（公司標準、協作規範、程式編碼原則、提交步驟等）置於 **internal_docs/**（倉庫根目錄，與 docs 同層）；產出此類內容時請依 `internal_docs/README.md` 與 `internal_docs/模組與程式代碼編碼原則.md` 等依循。
2. **新增「AI 產出審核」明文**  
   - 新增一條：**AI 輔助產出之 PRD、雛型、文件須經需求方（或專案負責人）審核後再合併至 main；涉及 01_Discovery、02_Analysis、docs、雛型之變更建議由需求方 Approve PR。**
3. **新增「程式與雛型命名」引用**  
   - 產出 PRD、雛型檔名、程式清單或 API/TABLE 命名時，請參照 **internal_docs/模組與程式代碼編碼原則.md**（主程式代號格式、子編碼、權限表原則、雛型檔名、禁止事項、檢查清單）。

### 4.2 高優先：開發團隊 AI 開發入口（銜接）

4. **建立「開發團隊 AI 開發指引」**  
   - 放置於 **internal_docs/**，檔名如 `開發團隊_AI開發指引.md`。  
   - 內容建議包含：  
     - **從需求方 Repo 取得什麼**：PRD（四項必備）、雛型、權限矩陣、欄位對照表、模組與程式代號架構。  
     - **AI 開發時必讀文件**：`internal_docs/模組與程式代碼編碼原則.md`、專案之「模組與程式代號架構」、PRD 內權限矩陣與情境說明。  
     - **產出程式/API/TABLE 時對齊規範**：主程式代號、子編碼格式、雛型檔名、TABLE/API 命名與程式代號對齊（摘要或連結至編碼原則 §6.3）。  
     - **子編碼清單**：建議開發團隊於專案內留存子編碼清單，並在程式文件中註明對應主程式代號。  
   - 開發方 Repo 可複製此指引，並在該 Repo 的 .cursorrules 或 README 中註明「AI 開發時請先閱讀 internal_docs/開發團隊_AI開發指引.md」。

5. **Handoff 格式（可選但建議）**  
   - 在 Tpl_GOV 或 新專案辦法中新增「**交付開發團隊時清單**」：交付時提供（1）PRD（含四項必備）、（2）雛型清單與檔名對照、（3）權限矩陣、（4）模組與程式代號架構、（5）internal_docs 內程式編碼原則與開發團隊 AI 開發指引之路徑或副本。  
   - 可製成 **Checklist** 供需求方勾選，確保開發團隊與其 AI 有單一、完整的輸入來源。

### 4.3 中優先：歸檔與索引

6. **將「新專案參考使用辦法_整合AI開發」歸檔至 internal_docs**  
   - 依 GOV_Inbox_歸檔審視與建議，執行歸檔後可讓新專案與開發團隊直接從 internal_docs 取得「整合 AI 開發」檢核與套用步驟。
7. **internal_docs/README 補上「AI 治理與開發銜接」**  
   - 在 internal_docs/README.md 本目錄文件一覽中，新增本文件（AI治理與開發銜接檢視與建議）及「開發團隊 AI 開發指引」（建立後）之連結，並簡述用途（AI 治理檢視、開發團隊 AI 開發入口）。

### 4.4 低優先：結構化與自動化

8. **PRD／權限矩陣之結構化（可選）**  
   - 若希望 AI 更容易解析，可在範本中約定固定 H2/H3 標題（如「## 模組與主程式代碼」「## 程式×角色權限矩陣表」），或於 YAML 中增加 `prd_sections: [module_codes, permission_matrix, scenarios, field_mapping]` 等標籤，供後續腳本或 AI 擷取。
9. **開發方 Repo 之 .cursorrules 片段範本（可選）**  
   - 於新專案辦法或 00_Templates 中提供「**給開發團隊用 .cursorrules 片段**」：包含必讀 internal_docs 路徑、程式編碼原則引用、產出程式/API 時對齊檢查，開發方建立新專案時可貼上並替換專案名稱與路徑。

## 5. 檢查清單（AI 治理與開發銜接自檢）

專案或新專案啟動時，可依下列勾選確認是否符合同意之 AI 治理與開發銜接水準：

- [ ] .cursorrules 已註明 internal_docs 用途及「對內文件」放置規則
- [ ] .cursorrules 已註明「AI 輔助產出須經需求方審核後再合併」或同等條款
- [ ] .cursorrules 或 internal_docs 已引用「模組與程式代碼編碼原則」，供 PRD／雛型／程式文件產出時依循
- [ ] internal_docs 已存在「開發團隊 AI 開發指引」（或同等 handoff 文件），且開發團隊已知曉並可複製至開發方 Repo
- [ ] 需求方交付開發團隊時，已提供 PRD（四項必備）、雛型清單、權限矩陣、程式代號架構，以及程式編碼原則／AI 開發指引之路徑或副本
- [ ] 新專案參考辦法（整合 AI 開發）已歸檔至 internal_docs，新專案可依其執行檢核與套用
- [ ] internal_docs/README 已列出 AI 治理檢視、開發團隊 AI 開發指引等文件，便於人員與 AI 發現
