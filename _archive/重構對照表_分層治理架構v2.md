## 一、目標結構

```
e-Consultant_SOP/
├── core/                 # 核心治理定義、母體文件、入口
├── standards/
│   ├── prd/              # PRD、需求、驗收、RFP 相關標準
│   ├── delivery/         # 交付包、子程式清單、工程師開工、需求釐清
│   ├── development/      # 雛型、TABLE/API、批次、cursorrules、子程式實作
│   ├── security/         # 資安標準
├── templates/            # 所有範本（原 00_Templates 內容）
├── ops/                  # 營運手冊、發布、提交、合併、檢查清單
├── projects/             # 專案工作區與專案特定產物（Discovery、Analysis、Inbox、docs 源）
└── README.md             # 倉庫根目錄總覽（新建或改寫）
```

## 二、分類原則摘要

| layer | 定義 | 典型內容 |
|-------|------|----------|
| **core** | 治理母體、公司級總則、全 Repo 入口與索引 | GOV 總則、公司專案標準規範、internal_docs 總覽、.cursorrules |
| **standards** | 可被專案引用的中央標準，依領域分子目錄 | prd / delivery / development / security |

| **ops** | 日常營運、發布、版控、解除卡關等操作指引 | OPS_*、提交/合併步驟、GitHub 檢查、MkDocs 發布 |
| **project** | 專案工作區、專案特定規格、對外網站源、待辦 | 01_Discovery、02_Analysis、_Inbox、docs、司機與廠商主檔規格等 |

## 四、風險與建議標註

### 4.1 重名風險

| 項目 | 說明 |
|------|------|
| **core/README_internal_docs.md** | 與根目錄 `README.md` 區分，已用不同檔名避免覆蓋。 |
| **standards/00_總覽.md** | 與 `core/` 下總覽職責不同；無重名，僅需在 00_總覽 內更新指向 `standards/prd/`、`standards/delivery/` 等連結。 |
| **多個 README** | `templates/README.md`、`projects/01_Discovery/README.md` 等分屬不同目錄，無重名風險。 |

### 4.2 是否需要拆分

| 原路徑 | 建議 |
|--------|------|
| internal_docs/index.md | 可維持單檔；若 v2 後 core 文件過多，可拆成「生命週期」「公司標準」「AI 與產出」等數個索引檔，再由 core/README 彙總。 |
| internal_docs/standards/00_總覽.md | 建議不拆；改為依 domain（prd/delivery/development/security）分區塊列出並連結至新路徑。 |
| GOV_e化專案治理總則.md | 母體文件建議不拆；可於文內增加「與分層架構對照」小節，指向 core/、standards/、ops/。 |

### 4.3 是否需要補 metadata

- **已有 YAML frontmatter**：多數 `internal_docs` 與 `internal_docs/standards` 下檔案已具 `title, type, status, tags, version, owner, last_updated`，僅需依新路徑補 `layer` 或 `domain` 標籤（若採用）。
- **需補 metadata**：`00_Templates` 下範本、根目錄與 `ops` 下部分檔案（如 OPS_*、提交變更、UNSTUCK）建議補 frontmatter，以利檢索與治理。
- **根 README.md**：新建或改寫時建議加上 `type: Index`、`layer: root`、以及對 core/、standards/、templates/、ops/、projects/ 的簡要說明與連結。

## 五、後續執行建議（不在此次執行）

1. **依表單執行搬移**：依「原路徑 → 新路徑」批次搬移，並以 git 保留歷史（如 `git mv`）。
2. **更新所有內部連結**：在 core、standards、templates、ops、projects 內將舊路徑改為新路徑；特別注意 `internal_docs/`、`00_Templates/`、`standards/` 的交叉引用。
3. **修正 mkdocs 與 CI**：若 `mkdocs.yml` 或 `.github/workflows/publish.yml` 引用 `docs/` 或特定路徑，需一併更新（或維持 `docs/` 在根目錄而僅將「來源」視為 projects 產出）。
4. **補齊 metadata**：對標記「建議」補 metadata 的檔案補上 frontmatter。
5. **撰寫根目錄 README.md**：說明 v2 分層架構、各層用途、以及從「生命週期」或「角色」出發的導覽。
