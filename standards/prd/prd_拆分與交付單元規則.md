# 中央標準｜PRD 拆分與交付單元規則

本文件定義**AI 驅動開發**下之文件層級、PRD 拆分原則與交付單元邊界。工程團隊可先取得**主程式架構與說明**，後續再依**主程式＋子程式**逐一取得 PRD 進行開發。

---

## 1. 設計原則

| 原則 | 說明 |
| :--- | :--- |
| **單一程式可獨立運行** | 每個主程式設定為可獨立運行的功能邊界，便於分開交付與測試 |
| **PRD 最小化** | 每個 PRD 僅涵蓋一個交付單元（主程式＋其子程式），不寫整個專案 |
| **先架構後細部** | 先建立主程式代號與說明，供工程師規劃；PRD 細部後續補上 |
| **RFP 保留系統全貌** | RFP 仍涵蓋系統全貌、階段、主程式清單；PRD 則分開撰寫 |
| **拆分最小邏輯** | 採用**最小可交付單元**拆分，有助於分給工程團隊**平行開發**，縮短整體交付時程 |

### 1.1 平行開發效益

採用本拆分原則時，可達成：

| 效益 | 說明 |
| :--- | :--- |
| **平行開發** | 不同主程式（如出口、進口、轉運）可分配不同工程師同時開發，互不阻塞 |
| **分開交付** | 主程式 A 完成即可先交付，不需等待主程式 B、C |
| **風險分散** | 單一主程式變更不影響其他主程式之開發與測試 |
| **需求穩定** | 每份 PRD 範圍小，需求變更影響範圍可控 |

**提醒**：需求規劃與情境流程審核建立歸檔時，建議採用此**拆分最小邏輯**，有利於後續分給工程團隊平行開發。

## 2. 文件層級與產出順序

### 2.1 文件層級關係

```
RFP／系統全貌
    ├── 階段劃分（Phase 1、Phase 2…）
    ├── 主作業分類（出口、進口、轉運…）
    └── 主程式架構（程式代號＋說明）  ← 工程師先取得
            │
            ├── PRD_主程式A＋子程式  ← 交付單元 1
            ├── PRD_主程式B＋子程式  ← 交付單元 2
            └── PRD_主程式C＋子程式  ← 交付單元 3
```

### 2.2 產出順序

| 順序 | 產出 | 說明 | 交付對象 |
| :--- | :--- | :--- | :--- |
| **1** | **RFP／系統全貌** | 系統目標、階段、主作業分類、主程式清單（代號＋簡述） | 客戶確認、工程師預覽 |
| **2** | **主程式架構文件** | 主程式代號、程式名稱、所屬作業、簡要說明；**子程式預留**（主檔、參數、選項維護等） | **工程師先取得** |
| **3** | **PRD（分開交付）** | 每個主程式＋其子程式為一 PRD；四項必備、雛型／批次規格、驗收條件 | 依交付單元逐一提供 |

## 3. 主程式與子程式定義

### 3.1 主程式

- **定義**：對應權限表、可獨立進入之作業入口；依 [程式編碼原則｜共用規範](../程式編碼原則_共用規範.md) 編列（x00 目錄、x10 建檔、x20 查詢、x30 歷史）。
- **邊界**：一個主程式 = 一個可獨立運行的功能單元。
- **範例**：CUS100 出口目錄、CUS110 出口建檔、CUS200 進口目錄、CUS210 進口建檔。

### 3.2 子程式

- **定義**：主程式所需之**主檔、參數、選項**維護，或建檔內之**子作業**（成單、審核、帳務等），或**批次程式**（查核、變更狀態、資料同步）。
- **延伸時機**：當主程式需要獨立維護畫面（如客戶主檔、供應商主檔、參數設定、選項代碼）時，延伸子程式；當有批次程式對應該主程式之查核或變更狀態時，延伸 `_batch_` 子程式。
- **編碼**：依 [程式編碼原則](../程式編碼原則_共用規範.md) §4，子程式可為 `主程式代號_識別`、`主程式代號_batch_<功能>`（批次，對應查核或變更狀態）；全域批次採 `BATCH001` 等（§4.6）。API／TABLE 命名見 §5。
- **納入 PRD**：子程式與其主程式**同一交付單元**，寫入同一份 PRD；對應主程式之批次程式亦納入該主程式之 PRD。

### 3.3 交付單元

| 交付單元 | 涵蓋範圍 | PRD 範圍 |
| :--- | :--- | :--- |
| **單元** | 1 個主程式 + 其子程式（主檔、參數、選項維護） | 1 份 PRD |
| **範例** | CUS110 出口建檔 + 其所需主檔維護（若獨立為子程式） | PRD_出口建檔_CUS110 |
| **範例** | CUS010 客戶主檔與聯絡人（全域，可獨立交付） | PRD_客戶主檔_CUS010 |

## 4. 拆分規則（AI 須遵守）

### 4.1 PRD 不寫整個專案

- ❌ 一份 PRD 涵蓋出口、進口、轉運全部
- ✅ 一份 PRD 僅涵蓋一個主程式＋其子程式

### 4.2 主程式架構先建立

- 在 RFP 或「主程式架構」文件中，**先**列出所有主程式代號與說明。
- 工程師可依此規劃選單、權限、技術架構。
- 子程式可於主程式架構中**預留**（如「CUS110 建檔，子程式：供應商主檔、運務主檔，待 PRD 補齊」）。

### 4.3 PRD 依交付單元分開撰寫

- 每個交付單元 = 1 份 PRD。
- PRD 檔名建議：`YYYY-MM-DD_PRD_<主程式代號>_<簡述>.md`
- 範例：`2026-02-12_PRD_CUS110_出口案件建檔.md`、`2026-02-12_PRD_CUS010_客戶主檔與聯絡人.md`

### 4.4 主檔、參數、選項的歸屬

| 類型 | 歸屬 | 說明 |
| :--- | :--- | :--- |
| **全域主檔** | 獨立主程式（如 CUS010） | 多個主程式共用，可獨立交付 |
| **單一主程式用主檔** | 該主程式之 PRD | 僅該主程式使用，納入其 PRD |
| **參數、選項** | 該主程式之 PRD 或獨立子程式 | 若維護畫面獨立，可為子程式 |

## 5. 主程式架構文件格式

工程師先取得之「主程式架構」應含：

| 章節 | 內容 |
| :--- | :--- |
| **系統全貌** | 階段、主作業分類、模組代號 |
| **主程式清單** | 程式代號、程式名稱、所屬作業、簡要說明 |
| **子程式預留** | 已知或預期之子程式（主檔、參數、選項），可標「待 PRD 補齊」 |
| **交付順序建議** | 建議之開發／交付順序（依相依關係） |

範例見專案內之「模組與程式代號架構」文件（如 `02_Analysis/2026-02-09_報關管理模組與程式代號架構.md`）。

## 6. RFP 與 PRD 分工

| 文件 | 涵蓋範圍 | 細度 |
| :--- | :--- | :--- |
| **RFP** | 系統全貌、階段、主作業、主程式清單（代號＋簡述）、預期效益、範圍 | 架構層級 |
| **主程式架構** | 主程式代號、名稱、所屬作業、子程式預留、交付順序 | 架構層級 |
| **PRD** | 單一主程式＋子程式；四項必備、功能、欄位、雛型／批次規格、驗收 | 實作層級 |

## 7. AI 系統設計建議時的要求與提醒

當 AI 提供**系統設計建議**或**需求規劃建議**時，須主動納入以下提醒：

### 7.1 拆分原則建議

- **主動建議**：需求規劃可採用**拆分最小邏輯**（主程式＋子程式為一交付單元），有助於分給工程團隊**平行開發**。
- **提醒時機**：產出 RFP、主程式架構、情境流程、PRD 或需求歸檔時。
- **建議用語**：  
  「建議採用 PRD 拆分與交付單元規則之**最小可交付單元**邏輯，將需求依主程式分開規劃，有利於工程團隊平行開發、分開交付。」

### 7.2 需求與情境流程審核建立歸檔時

在**需求與情境流程審核、建立、歸檔**時，AI 須提醒：

> 採用**拆分最小邏輯**（一份 PRD 僅涵蓋一個主程式＋其子程式）可有助於：
> - 分給工程團隊**平行開發**
> - 分開交付、縮短整體時程
> - 需求變更影響範圍可控
>
> 見 `internal_docs/standards/PRD_拆分與交付單元規則.md`。

### 7.3 系統設計建議檢查清單

AI 產出系統設計或需求規劃建議時，請自檢：

- [ ] 是否建議採用主程式架構先建立、PRD 分開撰寫？
- [ ] 是否提醒拆分最小邏輯有助於平行開發？
- [ ] 是否依主程式邊界劃分交付單元？
