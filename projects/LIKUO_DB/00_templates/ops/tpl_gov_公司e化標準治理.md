# [專案名稱] 公司 e化 標準治理

本文件為**公司標準 e化 專案治理**範本，整合：(1) **文件對內與對外**分類與規則、(2) **內部開發人員需完成之文件需求**、(3) **模組與程式命名規則**。新專案可複製本範本，替換 `[專案名稱]`、`[模組代號]` 等佔位符後使用。

**內外文件區分請先建立目錄架構**，再依下列規則產出與歸檔。

---

## 第零步：先建立目錄架構（內外區分）

在開始產出文件前，請先建立下列目錄。**對內文件請放在倉庫根目錄之 `internal_docs/`，不要放在 `docs/` 下**，以免 MkDocs 建置時將內部文件一併發布、造成權限外洩。

```
<專案根目錄>/
├── docs/                  # 僅對外：MkDocs 建置來源，僅放可對外內容
│   ├── index.md           # 或 project_overview.md：對外首頁
│   └── images/            # 對外網站用圖片
├── internal_docs/          # 對內：公司標準、協作規範、程式編碼、提交步驟等（不納入 MkDocs）
└── …其他目錄（00_Templates、02_Analysis 等）
```

| 目錄 | 對內/對外 | 說明 |
| :--- | :--- | :--- |
| **docs/** | 對外 | 專案總攬、操作手冊、SOP、雛型連結；**MkDocs 僅建置此目錄**，故僅可對外內容放此 |
| **internal_docs/**（倉庫根目錄，與 docs 同層） | 對內 | 規範、Runbook、開發指引等；**不在 docs 下**，MkDocs 不會掃到，避免權限被開放出去 |

**雛型 HTML、專案總攬草稿**：若需對內暫存，可置於 `internal_docs/prototype/` 或專案約定路徑，勿放於 `docs/` 下以免被建置。

## 第一部分：文件對內與對外 (DOCS)

### 1.1 文件分類與目錄

| 對象 | 目錄 | 是否對外發布 | 用途 |
| :--- | :--- | :--- | :--- |
| **使用者／客戶** | `docs/` | ✅ 是（如 MkDocs 首頁） | 系統說明、作業模式、程式清單與雛型連結、上線後操作手冊與 SOP |
| **開發團隊** | `internal_docs/`（倉庫根目錄） | ❌ 否 | 程式編碼原則、協作與分支、專案管理、提交步驟、文件撰寫指引、設計與 PRD 索引；雛型草稿可放此目錄下 |

**判斷原則**：內容是否會給**客戶或 end-user** 看？  
→ **是** → 對外（`docs/`）；**否**（專案管理、設計、開發流程、內部規範）→ 對內（**`internal_docs/`**，與 docs 同層，**不放在 docs 下**，避免 MkDocs 開放權限）。  
對內目錄**不納入 MkDocs 建置**，不對外發布、不對外搜尋。

---

### 1.2 對外文件（使用者）規則

**放置位置**：`docs/project_overview.md`（對外網站首頁即此檔），以及操作手冊、SOP（可另建 `docs/manuals/` 等）。

**禁止出現**：

- 設計或分析文件之**路徑**（如 `02_Analysis/…`、`internal_docs/…`）
- 內部**狀態**（如 Draft、待規劃、開發中）
- 模組「規劃範圍」「待規劃」等**規劃狀態**描述
- 版本控制、分支、PR、審核流程等**開發流程**細節
- 檔案命名規則、儲存位置等**實作細節**

**可出現**：

- 系統與專案**目標**、**作業模式**、**程式清單**（表格：程式代號、程式名稱、用途、雛型連結）
- **雛型**：僅以**連結**呈現（不寫實體路徑），例：`[雛型檢視](prototype/YYYY-MM-DD_prototype_<程式代號>.html)`
- 文件**版次**與**發布日期**（表格式即可）
- 完成後將提供**操作手冊、SOP、教育訓練**等說明

---

### 1.3 對內文件（開發團隊）規則

**放置位置**：**倉庫根目錄**之 `internal_docs/` 下所有 Markdown 與子目錄（勿放在 `docs/` 下，以免被 MkDocs 建置出去）。

**可出現**：

- 設計文件、PRD、雛型規範之**路徑**與**索引**
- 文件或專案**狀態**（Draft、已發布、待規劃）
- **程式編碼原則**、雛型 HTML 命名與發布路徑
- **協作與治理**、分支、PR、提交步驟
- **專案管理**文件與紀錄（會議、決策、變更歷程）
- 如何要求文件優化、撰寫範本等**產出指引**

**歸檔**：專案管理或開發過程產生的紀錄應放在對內目錄，並在對內首頁（如 `internal_docs/README.md`）列出，避免與對外首頁混在一起。

---

### 1.4 產出時自檢（對內 vs 對外）

| 情境 | 應放置 | 內容注意 |
| :--- | :--- | :--- |
| 給客戶看的總覽、程式清單、雛型入口 | `docs/` | 無路徑、無狀態、雛型僅連結 |
| 操作手冊、SOP、教育訓練教材 | `docs/`（或 docs 下對外子目錄） | 同對外規則 |
| 程式代號、編碼原則、雛型命名規範 | `internal_docs/`（倉庫根目錄） | 可有路徑與範例 |
| 協作、分支、PR、提交步驟 | `internal_docs/` | — |
| 雛型 HTML、專案總攬草稿 | `internal_docs/`（如 `internal_docs/prototype/`） | 不放在 docs 下，不發布 |
| 設計文件、PRD 本體 | 維持 `02_Analysis/` 等，在 `internal_docs/` 做**索引** | 對外文件不直接連結 |

## 第二部分：內部開發人員需完成的文件需求

需求方提供各功能作業之 PRD 或新文件需求時，須依下列**標準產出要求**產出，作為開發方**框架、TABLE、API 命名與權限設計**之依據；開發方則須依 PRD 與雛型實作，並遵守**開發要求（含資安）**。

### 2.1 產出項目（四項必備）

| 項目 | 說明 | 範本／參考 |
| :--- | :--- | :--- |
| **模組與主程式代碼** | 該功能作業所屬模組及**完整**主程式代號清單（權限表用）；須**先建立完成**再撰寫 PRD。 | 本文件第三部分、各模組之程式代號架構文件 |
| **程式 × 使用者角色權限矩陣表** | 依情境產出：主程式代號 × 使用者角色之權限矩陣（可執行／可查詢／不可見等）。 | `Tpl_權限矩陣表.md` |
| **對應情境說明** | 各使用情境之操作者角色、可存取程式、流程與驗收說明。 | `Tpl_PRD_功能作業.md` 內「情境說明」章節 |
| **模組欄位名稱中英文對照表** | 該模組**所有客戶作業表單與報表**之欄位：**中文、英文、縮寫、用途**表列；與客戶對齊表單與報表名稱，新欄位時提醒**新增／對應／修改／刪除**，避免需求失真。 | `Tpl_模組欄位名稱對照表.md` |

### 2.2 產出順序與用途

1. **先建立**模組與主程式代碼（含目錄、建檔、查詢含增修、歷史歸檔等），再撰寫該功能作業 PRD。
2. **依情境**產出權限矩陣表與情境說明，與主程式代號對齊。
3. **同時建立或更新**該模組之**欄位名稱中英文對照表**，涵蓋該模組所有客戶作業表單之欄位；專案可能分開時段建置，此表可讓客戶與我方持續對齊表單與報表名稱。
4. 上述產出供開發方用於：**框架**、**資料表（TABLE）**、**API** 之命名與權限規劃，PRD 須一併提供或引用。

### 2.3 放置與歸檔

- 功能作業 PRD 本體：`02_Analysis/`（或專案約定之 PRD 目錄）。
- 權限矩陣表：可獨立成檔或併入 PRD 章節。
- **欄位名稱對照表**：建議獨立成檔（如 `02_Analysis/YYYY-MM-DD_模組名_欄位名稱對照.md`），該模組僅維護一份；有新表單或新欄位時更新同一檔並註記異動。
- 程式代號架構：`02_Analysis/` 或對內目錄，並在對內首頁索引中列出。

### 2.4 開發要求（含資安基本提醒）

開發方於**規劃與實作**時，除依 PRD 與雛型實作外，須注意下列**資安基本要求**，並依公司或客戶既有資安規範從嚴辦理：

| 類別 | 提醒要點 |
| :--- | :--- |
| **身分認證與登入** | 密碼不可明文儲存；登入／Session 逾時與登出機制；若客戶或公司要求，考量多因素認證（MFA）。 |
| **授權與權限** | 依 PRD 之**權限矩陣表**與情境實作，採**最小權限**原則；前後端皆須驗證身分與權限，不得僅依前端隱藏按鈕。 |
| **敏感資料** | 個資、客戶資料之存取與儲存須受控；畫面與報表依角色遮罩或限制顯示；日誌與除錯輸出**不得**含密碼、完整個資或敏感欄位。 |
| **傳輸** | 對外或敏感連線使用 **HTTPS**；避免將敏感參數置於 URL 或 Cookie；API 與前後端通訊依環境要求加密。 |
| **日誌與稽核** | 重要操作（登入、權限變更、關鍵業務異動）可追溯；日誌內容避免記錄敏感資料；依公司或客戶規定保留與保護。 |
| **依循既有規範** | 依**公司資安政策**與**客戶資安／個資要求**（若有）規劃；若有弱掃、滲透測試或合規需求，預留時程與項目。 |

若有額外資安或合規需求，應寫入 PRD 或另附資安需求清單。

### 2.5 雛型設計時之資安建議（需求方／產出雛型者）

| 對應類別 | 雛型設計／生成時建議 |
| :--- | :--- |
| **授權與權限** | 依權限矩陣表與情境，在雛型中**區分角色**示範；標示「此按鈕／連結僅 XX 角色可見」；避免僅產出「全部功能可見」畫面。 |
| **敏感資料** | 雛型中之敏感欄位以**遮罩**（如 `***`）或**假資料**示範，**不使用真實**客戶或業務資料；若為範例，註記「僅供示意，正式環境不得使用真實個資」。 |
| **登入與登出** | 雛型若含登入／操作流程，建議保留**登出**按鈕或連結之示意，並可註記「逾時登出與 Session 管理由正式系統實作」。 |
| **傳輸與環境** | 可於說明或註記中註明「正式環境須以 **HTTPS** 提供」；表單送出行為可註記「實際送出須經後端驗證與加密」。 |

## 第三部分：模組與程式命名規則

本部分為**跨專案、跨模組**共用之程式編碼原則。各專案／模組須選定**模組代號**與**主作業分類（百位數）**後，依下列原則編列主程式與子編碼。

### 3.1 代號結構

| 組成 | 說明 | 範例 |
| :--- | :--- | :--- |
| **模組代號** | 二～三碼英文縮寫，同一專案／模組內不重複 | `CUS`、`KID` 等，由各專案訂定 |
| **功能代號** | 三位數，百位數對應主作業類別，十位個位對應功能類型 | `110`、`120` |
| **完整主程式代號** | `<模組代號><功能代號>`，用於**使用者授權權限表** | `CUS110`、`KID210` |

### 3.2 權限表主程式原則（每作業四支）

**僅下列四類**編列為主程式代號，登錄於使用者授權之權限表；對應選單／功能入口，供權限控管使用。

| 十位個位 | 功能類型 | 說明 |
| :--- | :--- | :--- |
| **x00** | 作業目錄 | 該類作業入口／選單 |
| **x10** | 案件建檔 | 案件新增與維護；其內流程步驟（成單、審核、帳務、結案等）為**子作業**，不另編主程式代號 |
| **x20** | 案件查詢（含增修） | 案件查詢與增刪修；若有匯出、統計需求，以子編碼延伸 |
| **x30** | 案件歷史歸檔 | 歷史歸檔與歷史資料查詢 |

- 每一**主作業**（由專案定義）對應一組百位數，並編列上述四支主程式（x00、x10、x20、x30）。
- **匯出、統計**等為程式延伸，不納入權限表主程式，以**子編碼**編列。

### 3.3 子編碼原則（不納入權限表）

下列項目**不**給予獨立主程式代號，由**程式設計師**依團隊規則以**子編碼**編列，不列入使用者授權權限表。

| 類型 | 編碼方式 | 範例 |
| :--- | :--- | :--- |
| 建檔內子作業 | `<主程式代號>_<識別>` | `CUS110_Order`、`CUS110_DocAudit` |
| 報表、開窗、查詢子畫面 | `<主程式代號>_<識別>` | `CUS110_Report01`、`CUS120_Query01` |
| 匯出、統計（程式延伸） | `<主程式代號>_<識別>` | `CUS120_Export01`、`CUS120_Report01` |

**編列權責**：**主程式代號**（權限表用）由**需求方／專案**訂定；**子編碼**由**程式設計師**依本規範與團隊規則編列。

### 3.4 百位數規劃（各專案自訂）

- **百位數**由各專案／模組自訂，對應該專案之**主作業分類**（例如：1xx 作業 A、2xx 作業 B…）。
- 同一專案內，同一功能的十位個位意義需一致（x00 皆為目錄、x10 皆為建檔、x20 皆為查詢含增修、x30 皆為歷史歸檔）。
- 新增主作業時，以未使用之百位數遞增編列。

### 3.5 雛型與文件命名

| 項目 | 規則 |
| :--- | :--- |
| **雛型 HTML 檔名** | `YYYY-MM-DD_prototype_<程式代號>.html`（例：`2026-02-05_prototype_CUS100.html`） |
| **文件與程式** | 文件、雛型、程式命名與程式代號保持一致 |
| **作廢／刪除** | 若對外有編號，建議採作廢保留編號、系統單號重取，避免空號 |

### 3.6 專案套用方式

1. **選定模組代號**：如 `[模組代號]`（由專案訂定）。
2. **訂定主作業分類與百位數**：列出該模組之所有主作業，並分配百位數（1xx、2xx、3xx…）。
3. **產出主程式代號清單**：每一主作業編列四支主程式（目錄、建檔、查詢含增修、歷史歸檔），寫入該專案之架構或 PRD 文件。
4. **權限表**：僅登錄上述主程式代號。
5. **子編碼**：由程式設計師於開發時依 §3.3 編列，並於專案內紀錄。
