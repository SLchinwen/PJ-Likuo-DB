# 如何更新對外文件並重新提供給客戶

對外網站**首頁即專案總攬**（`docs/index.md`）。更新該檔後合併進 **main**，網站會自動重新發布，客戶即可看到最新內容。

---

## 一、更新內容時請遵守

- **不寫**設計文件路徑、內部狀態、待規劃等；**可寫**版次、發布日期、雛型連結。詳見 [文件治理規範](docs_治理規範.md)。

---

## 二、操作步驟

### 1. 修改對外文件

在專案中編輯 **`docs/index.md`**（對外網站僅此頁）：系統目標、作業模式、程式清單（表格）、完成後提供項目、文件版次。

若新增雛型：在專案總攬的**程式清單**表格「雛型」欄補上連結，例如：`[雛型檢視](prototype/YYYY-MM-DD_prototype_CUS110.html)`。

### 2. 提交並推送到 main

**目前 main 未設分支保護**，可直接提交並推送到 main：

1. 確認在 **main** 分支且已拉取最新：`git checkout main` → `git pull origin main`
2. 暫存變更：`git add docs/index.md`（或您修改的檔案）
3. 提交：`git commit -m "docs: 更新對外文件（簡短說明）"`
4. 推送：`git push origin main`

若日後改為 main 受保護，須改透過分支與 PR，見 [提交變更步驟](提交變更步驟.md)。

### 3. 自動發布

合併到 **main** 後，GitHub Actions 會自動：

- 執行 `mkdocs build`
- 部署到 **GitHub Pages**（gh-pages 分支）

客戶端：重新整理手冊網址（如 `https://slchinwen.github.io/HER_CHYUN/`）即可看到更新後內容，**無需另寄檔案**。

---

## 三、若需主動通知客戶

發布完成後可：

- 傳送**手冊網址**與**本次更新摘要**（例如：更新程式清單、新增 CUS110 雛型連結、版次 v1.1）給客戶；
- 或依貴公司習慣以郵件／會議說明「對外文件已更新，請重新整理頁面查閱」。

---

## 四、摘要

| 步驟 | 說明 |
| :--- | :--- |
| 1. 編輯 | 改 `docs/index.md` 或 `docs/index.md`，遵守對外文件規則 |
| 2. 提交 | 在 main 上 commit → push origin main（目前無分支保護） |
| 3. 發布 | 合併後自動建置並部署至 GitHub Pages |
| 4. 客戶 | 重新整理手冊網址即看到最新版；可另通知更新摘要 |
