# 開發團隊 AI 開發指引

本文件為**開發團隊**在使用 **Cursor／AI 輔助開發**時之入口指引：從需求方 Repo 取得什麼、必讀哪些內部文件、產出程式／API／TABLE 時如何對齊公司規範。需求方交付時可一併提供本指引路徑或副本，供開發方 Repo 複製使用。

---

## 1. 從需求方 Repo 取得什麼

開發團隊（含 AI 輔助）應以需求方 Repo 為**單一需求來源**，對齊其版次或 Tag。建議取得並常駐參照下列產物：

| 產物 | 說明 | 用途 |
| :--- | :--- | :--- |
| **PRD（四項必備）** | 模組與主程式代碼、程式×角色權限矩陣表、對應情境說明、模組欄位名稱中英文對照表 | 框架、TABLE、API 命名與權限設計依據 |
| **雛型清單與檔名** | 雛型 HTML 檔名格式：`YYYY-MM-DD_prototype_<程式代號>.html` | 畫面與程式代號對齊、權限對應 |
| **模組與程式代號架構** | 主作業百位數、主程式代號清單（x00/x10/x20/x30） | 不可自創主程式代號；子編碼由開發方依規範編列 |
| **internal_docs 路徑或副本**（專案 Repo 內） | 程式編碼原則、協作與治理（若需求方提供） | 命名、權限、資安與交付節奏 |

---

## 2. AI 開發時必讀文件（建議載入或複製至開發方 Repo）

| 文件 | 說明 |
| :--- | :--- |
| **模組與程式代碼編碼原則** | 本 Template Repo 為 `core/模組與程式代碼編碼原則.md`；專案 Repo 內為 `internal_docs/模組與程式代碼編碼原則.md`（或需求方提供之副本）。主程式／子編碼格式、權限表原則、雛型檔名、TABLE/API 對齊、禁止事項、**程式文件生成檢查清單**。 |
| **專案之模組與程式代號架構** | 需求方 Repo 之 02_Analysis 或 internal_docs（專案 Repo 內）架構文件。主程式代號為權限表與命名依據，不得擅自新增。 |
| **PRD 內權限矩陣與情境說明** | 各功能作業 PRD 之權限矩陣表與情境說明，實作時依最小權限與情境驗收。 |

開發方 Repo 若使用 .cursorrules，建議註明：**產出程式、API、TABLE 或程式文件時，請參照「模組與程式代碼編碼原則」與專案之程式代號架構。**

---

## 3. 產出程式／API／TABLE 時對齊規範（摘要）

以下摘要自「模組與程式代碼編碼原則」，詳細請以該文件為準。

| 產出類型 | 對齊要點 |
| :--- | :--- |
| **主程式代號** | 僅需求方訂定；格式 `<模組代號><3位數>`（如 CUS110）；權限表僅登錄 x00/x10/x20/x30。 |
| **子編碼** | 格式 `<主程式代號>_<識別>`（如 CUS110_Order、CUS120_Export01）；由程式設計師編列，不登錄權限表；建議專案內留存子編碼清單。 |
| **雛型檔名** | `YYYY-MM-DD_prototype_<主程式代號>.html`，與權限表一致。 |
| **框架／路徑** | 可依主程式代號或模組分層，例：CUS/110、CUS110，與權限表對齊。 |
| **TABLE** | 建議以業務意義命名，設計文件註明對應之主程式代號或模組；可採專案約定前綴＋業務名稱。 |
| **API** | 路徑或代碼含模組／程式識別，例：`/api/cus/110/...` 或 `CUS110_Query`，與主程式代號對齊，便於權限與稽核。 |

產出或審核程式相關文件時，請依 **模組與程式代碼編碼原則 §9 程式文件生成時之檢查清單** 勾選。

---

## 4. 資安與交付（提醒）

- **開發階段資安**：依需求方提供之「協作與治理」或「開發要求」§2.5（身分認證、授權與最小權限、敏感資料、傳輸、日誌與稽核）。
- **交付與同步**：開發方在**自有專案**實作，不於需求方 Repo 提交程式碼；每週與需求方對齊進度與測試修正；客戶回饋由需求方通知後再調整。
- **AI 產出**：開發方使用 AI 產出之程式或文件，仍須經專案約定之審核（如 Code Review、需求方驗收）後再合併或交付。
