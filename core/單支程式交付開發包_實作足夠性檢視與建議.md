# 單支程式交付開發包｜實作足夠性檢視與建議

本文件以**AI 協助開發**與**資深工程師**兩種角色視角，檢視目前治理規範與「單支程式交付開發包」內容是否**足夠進行實作開發**，並提出缺口與改進建議。供專案負責人與需求方產出、審核交付包時參考。

---

## 1. 檢視目的與角色視角

| 角色 | 檢視重點 |
| :--- | :--- |
| **AI 協助開發** | 交付包是否含結構化、可解析的規格（程式代號、欄位、API、驗收）；AI 能否依子程式清單拆任務、產 schema/API、對齊命名；缺項時能否從雛型/PRD 推論或提醒補齊。 |
| **資深工程師** | 是否足以開工（環境、依賴、技術決策）；是否足以設計與實作（TABLE/API、權限、錯誤處理、邊界條件）；是否足以驗收（可執行之驗收條件、測試情境）。 |

---

## 2. 治理規範與交付包內容盤點

### 2.1 目前交付包結構（有 UI）

依 [交付包結構範本](standards/交付包結構範本.md)：

| 區塊 | 內容 | 用途 |
| :--- | :--- | :--- |
| **README.md** | 交付說明、版次、對應 Tag、本階段範圍、取得更新方式 | 開工前脈絡 |
| **.cursorrules** | 公司標準精華 + 本專案程式代號、欄位、雛型路徑 | AI 行為與引用 |
| **01_規格/** | PRD 摘要、程式代號與權限矩陣、欄位對照表、情境與驗收說明、開發要求與資安清單、**子程式清單** | 實作依據 |
| **02_雛型/** | prototype/*.html、操作步驟說明 | 畫面與流程對照 |
| **03_附錄/** | 工程師開工指引、流程圖與架構、（選用）schema_參考、API_規範 | 開工步驟、技術產物 |

### 2.2 治理規範對齊

- **PRD 四項必備**（模組與主程式代碼、權限矩陣、情境說明、欄位對照表）→ 交付包內以「摘要＋權限＋欄位＋情境與驗收」形式涵蓋。
- **子程式清單**、**欄位對照表**：產出交付包時必備；可 AI 自動產出。
- **工程師開工指引**：必備；含建議步驟、AI Prompt、環境、準備事項。
- **TABLE/API**：依 [TABLE與API設計_AI產出指引](standards/TABLE與API設計_AI產出指引.md) 可產出至 `03_附錄/schema_參考.md`、`API_規範.md`，但範本中為**選用**。

## 3. AI 協助開發視角：是否足夠？

### 3.1 足夠之處

| 項目 | 說明 |
| :--- | :--- |
| **單一主程式邊界** | 交付單元＝主程式＋子程式，範圍清楚，利於 AI 只讀此包即可展開任務。 |
| **子程式清單** | 有清單＋實作建議標準（類型、順序、命名、API pattern），AI 可拆解為工作包與產出順序。 |
| **欄位對照表** | 中英、縮寫、用途一致，利於產出 schema、API 請求/回應、表單欄位。 |
| **權限矩陣** | 程式×角色明確，利於 API 權限註記與前端顯示邏輯。 |
| **雛型＋操作步驟** | 可對應 data-screen/data-step、按鈕與開窗，減少「做什麼」的歧義。 |
| **.cursorrules** | 交付包內嵌專案程式代號、路徑，AI 產出時可對齊命名與引用。 |

### 3.2 不足或風險

| 項目 | 說明 |
| :--- | :--- |
| **schema / API 是否在包內** | 結構範本中 schema_參考、API_規範為附錄選用。若**未產出**，AI 需從欄位對照表＋子程式清單**自行推論**，易與既有系統或團隊慣例不一致。建議：**有 UI 交付包建議預設產出** schema_參考與 API_規範（至少本程式範圍），並在開工指引中註明「可依此擴充」。 |
| **驗收條件結構化程度** | 情境與驗收說明若為自然段、未依驗收條件範本（ID、類型、檢核方式），AI 難以自動產出測試案例或檢核清單。建議：交付包內**驗收條件**盡量採表格（ID、類型 F/P/S/D/B、描述、檢核方式、對應規格）。 |
| **依賴與前置程式** | 若本程式依賴他程式（如客戶主檔、船期 API），僅在開工指引「準備事項」提及可能不足。建議：交付包 README 或 01_規格 中**明確列出依賴程式／API／TABLE**，以及「無此依賴時可 mock 或暫不實作」之說明。 |
| **錯誤與邊界情境** | PRD 摘要與情境若偏重 happy path，AI 易忽略驗證規則、錯誤訊息、權限拒絕等。建議：欄位對照表或情境中**至少列出必填、格式、長度、錯誤情境**；驗收條件含 S（資安）、D（資料）類型。 |
| **範本實體存在與否** | 交付包結構範本引用多個 Tpl_*_交付用（如 Tpl_PRD_本階段範圍摘要、Tpl_情境與驗收說明_交付用、Tpl_子程式清單_交付用、Tpl_工程師開工指引）。若本 Template Repo 之 templates/（或專案 Repo 之 00_Templates）中**未齊備**，產出品質與一致性依賴人工或臨場發揮。建議：**補齊交付用範本**或於 AI 產出指引中註明「若無範本則依中央標準章節產出」。 |

## 4. 資深工程師視角：是否足夠？

### 4.1 足夠之處

| 項目 | 說明 |
| :--- | :--- |
| **範圍邊界** | 單支主程式＋子程式，不夾雜他程式，估工與排程清楚。 |
| **權限與角色** | 權限矩陣表可直接對應後端權限設計與前端選單/按鈕顯示。 |
| **雛型與操作步驟** | 可對照畫面與流程，減少需求再確認。 |
| **開工指引** | 有建議步驟、環境、準備事項，可快速開工。 |
| **編碼原則引用** | 程式代號、子編碼、TABLE/API 命名有中央標準可依，減少命名歧異。 |

### 4.2 不足或風險

| 項目 | 說明 |
| :--- | :--- |
| **技術棧與介面契約** | 若未在交付包或開工指引中註明「本專案前端/後端/DB/認證」決策，工程師需自行假設或再問。建議：**技術棧與認證**（含 API 風格、錯誤碼格式）於 README 或 03_附錄 簡要固定，或註明「依專案技術棧決策備忘」。 |
| **環境與帳號** | 開工指引提到 Azure 等，但**實際連線、帳號、權限**通常由專案另行提供。建議：交付包內僅寫「環境需求清單」與「取得方式」（如聯絡誰、連結至 internal_docs/Azure環境建議），避免寫死機密。 |
| **依賴程式之介面** | 依賴他程式時，若無「呼叫介面摘要」（API 路徑、必填參數、回傳格式），工程師需翻他份交付包或問人。建議：**依賴清單**＋**介面摘要**（或連結至他程式 API_規範）放入 03_附錄 或 README。 |
| **可執行驗收** | 驗收條件若只寫「應能新增案件」，未寫「以承辦登入→填 A/B/C→送出→列表可見」等可執行步驟，驗收時易有認知落差。建議：**關鍵情境**對應到「操作步驟＋預期結果」，與驗收條件 ID 對應。 |
| **批次／非同步** | 若主程式含「匯出」「列印」「非同步處理」，交付包是否含觸發條件、逾時、失敗重試？目前批次規格在無 UI 交付包較完整；有 UI 內嵌之批次行為建議在情境與驗收中**至少點出**。 |

## 5. 缺口彙總與優先建議

### 5.1 缺口彙總

| 缺口 | 影響對象 | 說明 |
| :--- | :--- | :--- |
| schema_參考 / API_規範 為選用，常未產出 | AI、工程師 | 實作時需自行推論，易與既有系統不一致。 |
| 驗收條件未結構化（缺 ID、類型、檢核方式） | AI、工程師、驗收 | 難以對應測試案例與自動檢核。 |
| 依賴程式／API 未在包內明列或摘要 | 工程師、AI | 開工與實作時需額外查詢或假設。 |
| 錯誤與邊界情境不足 | AI、工程師 | 易忽略驗證、錯誤訊息、權限拒絕。 |
| 交付用範本未齊備 | 需求方、AI | 產出品質與一致性不穩定。 |
| 技術棧／認證未在包內固定或引用 | 工程師 | 需再確認，延遲開工。 |

### 5.2 改進建議（按優先順序）

| 優先 | 建議 | 作法 |
| :--- | :--- | :--- |
| **P0** | **有 UI 交付包建議預設產出 schema_參考與 API_規範** | 於 [交付包結構範本](standards/交付包結構範本.md) 將 `03_附錄/schema_參考.md`、`API_規範.md` 列為**有 UI 時建議必備**；[AI_產出交付文件指引](AI_產出交付文件指引.md) Step 6 後增加「Step 6.5：依欄位對照表與子程式清單產出 schema_參考與 API_規範（本程式範圍）」。 |
| **P0** | **驗收條件結構化** | 產出交付包時，情境與驗收說明內之驗收條件**依 [驗收條件範本](standards/驗收條件範本.md)** 產出（ID、類型 F/P/S/D/B、描述、檢核方式、對應規格）；可併入 `情境與驗收說明.md` 或獨立 `驗收條件_本階段.md`。 |
| **P1** | **交付包內明列依賴與介面摘要** | README 或 01_規格 增加「依賴程式／API／TABLE」清單；若有依賴，03_附錄 增加「依賴介面摘要」（路徑、必填參數、回傳要點）或連結至他程式 API_規範。 |
| **P1** | **錯誤與邊界情境** | 欄位對照表或情境說明中，至少列出必填、格式/長度、錯誤情境（如「統編格式錯誤」「權限不足」）；驗收條件含 S、D 類型。 |
| **P1** | **補齊交付用範本** | 於本 Template Repo 之 templates/（或專案 00_Templates）建立 tpl_交付開發包_readme、tpl_prd_本階段範圍摘要、Tpl_情境與驗收說明_交付用、Tpl_子程式清單_交付用、Tpl_工程師開工指引 等，與交付包結構範本對齊；或於 AI 產出指引註明「無範本時依標準章節產出」。 |
| **P2** | **技術棧與認證在包內固定或引用** | 交付包 README 或 03_附錄 註明「本專案技術棧與認證依 [技術棧與認證_決策備忘](技術棧與認證_決策備忘.md)」或簡要列出；減少開工時再確認。 |
| **P2** | **開工前自檢清單** | 於 [工程師開工指引｜治理規範](standards/工程師開工指引_治理規範.md) 或範本中增加「交付包接收者自檢」：依賴是否取得、schema/API 是否在包內、驗收條件是否可執行。 |

## 6. 結論

- **整體**：目前治理規範與單支程式交付開發包**已具備實作所需之主幹**（範圍、權限、欄位、子程式、雛型、開工指引），在**規格齊備且依範本產出**的前提下，對 AI 與資深工程師**可達「基本足夠」**。
- **關鍵補強**：將 **schema_參考／API_規範** 列為有 UI 交付包之建議必備、**驗收條件結構化**、**依賴與介面摘要**、**錯誤與邊界情境** 與 **交付用範本齊備**，可顯著提升「拿包即可開工、實作有據、驗收可檢核」之程度。
- **後續**：建議將本檢視結論納入 [交付前檢查與待補清單](交付前檢查與待補清單.md)，並於下次交付包產出時試行 P0／P1 建議，再依工程團隊回饋微調。
