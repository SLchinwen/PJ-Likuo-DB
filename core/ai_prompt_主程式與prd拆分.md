# AI｜Prompt 主程式與 PRD 拆分

本文件提供**與 AI 協作**時之 Prompt 範例，讓 AI 依 [PRD 拆分與交付單元規則](standards/PRD_拆分與交付單元規則.md) 產出主程式架構、分開的 PRD 與交付文件。

---

## 1. 使用前提醒

- 產出前請確認 AI 已讀取或可參考：本 Template Repo：`standards/PRD_拆分與交付單元規則.md`；專案 Repo：internal_docs/standards/
- 可於對話中附上該文件路徑，或於 .cursorrules 中引用

---

## 2. Prompt 範例

### 2.1 產出主程式架構（工程師先取得）

**情境**：系統全貌已有（RFP、表單整理），需產出主程式代號與說明，供工程師規劃。

```
請依「PRD 拆分與交付單元規則」產出主程式架構文件。

【輸入】
- 系統階段：[例：Phase 1 出口與客戶主檔、Phase 2 進口與轉運]
- 主作業分類：[例：出口、進口、轉運、機場、SP、船用品]
- 參考文件：[例：02_Analysis/2026-02-09_報關管理模組與程式代號架構.md、各作業表單整理]

【產出要求】
1. 主程式清單：程式代號、程式名稱、所屬作業、簡要說明
2. 子程式預留：各主程式可能延伸之主檔、參數、選項維護，標「待 PRD 補齊」若細部未定
3. 交付順序建議：依相依關係（如客戶主檔先於出口建檔）
4. 放置：02_Analysis/YYYY-MM-DD_主程式架構_專案名.md

請遵守：PRD 不寫整個專案；主程式架構先建立；後續 PRD 分開撰寫。
```

### 2.2 產出單一主程式之 PRD（分開交付）

**情境**：主程式架構已建立，需產出某主程式＋其子程式的 PRD。

```
請依「PRD 拆分與交付單元規則」產出 [主程式代號] 的 PRD。

【主程式】[例：CUS110 出口案件建檔]
【子程式】[例：無；或 供應商主檔、運務主檔 若需獨立維護]

【輸入】
- 主程式架構：internal_docs/standards/PRD_拆分與交付單元規則.md、02_Analysis/主程式架構文件
- 需求來源：[例：02_Analysis/2026-02-12_需求變更_出口作業、客戶雛型回饋]
- 欄位對照表：[例：該模組欄位名稱對照表]

【產出要求】
1. 一份 PRD 僅涵蓋此主程式＋其子程式（交付單元）
2. 四項必備：模組與主程式代碼、權限矩陣、情境說明、欄位對照表（可引用或摘錄）
3. 檔名：YYYY-MM-DD_PRD_<主程式代號>_<簡述>.md
4. 放置：02_Analysis/

請遵守：PRD 最小化；不寫其他主程式之細部；主檔、參數、選項若為此主程式專用則納入。
```

### 2.3 從系統全貌萃取主程式架構

**情境**：有 RFP 或系統全貌文件，需萃取並產出主程式架構。

```
請從 [文件路徑] 萃取主程式架構，產出「主程式架構」文件。

【輸入】
- RFP／系統全貌：[例：02_Analysis/系統全貌文件、各作業表單整理]
- 程式編碼原則：internal_docs/程式編碼原則_共用規範.md

【產出要求】
1. 列出所有主程式：程式代號、程式名稱、所屬作業、簡要說明
2. 依 x00 目錄、x10 建檔、x20 查詢、x30 歷史 結構編列
3. 子程式預留：各主程式可能需之主檔、參數、選項，標「待 PRD 補齊」
4. 交付順序建議
5. 放置：02_Analysis/YYYY-MM-DD_主程式架構_專案名.md

工程師將先取得此文件，後續再依主程式分開取得 PRD。
```

### 2.4 產出交付開發包（單一主程式範圍）

**情境**：某主程式 PRD 已確認、雛型已定稿，需產出該主程式的交付開發包。

```
請依「協同開發文件治理流程」與「PRD 拆分與交付單元規則」產出交付開發包。

【本階段範圍】僅 [主程式代號] 及其子程式
【不納入】其他主程式（如進口、轉運）

【輸入】
- PRD：02_Analysis/YYYY-MM-DD_PRD_<主程式代號>_xxx.md
- 雛型：docs/prototype/ 下該主程式之雛型
- 主程式架構：02_Analysis/主程式架構文件

【產出要求】
1. 交付包僅含本主程式＋子程式之規格、雛型、驗收
2. 依 本 Template Repo：core/AI_產出交付文件指引.md；專案 Repo：internal_docs/AI_產出交付文件指引.md 產出
3. 放置：可為 交付開發包/ 子目錄或壓縮包

請遵守：交付單元 = 主程式＋子程式；不納入其他主程式。
```

### 2.5 檢查 PRD 是否符合拆分規則

**情境**：產出 PRD 後，請 AI 自檢是否符合拆分規則。

```
請檢查 [PRD 文件路徑] 是否符合「PRD 拆分與交付單元規則」。

【檢查項目】
1. 是否僅涵蓋單一主程式＋其子程式？（不應涵蓋整個專案）
2. 主程式代號是否與主程式架構一致？
3. 主檔、參數、選項是否正確歸屬（全域 vs 該主程式專用）？
4. 四項必備是否齊備？

請列出符合／不符合項目，若有不符合請建議修正。
```

## 3. 簡短指令（可加入 .cursorrules）

可於 .cursorrules 或對話開頭加入：

```
產出 PRD 時須遵守：internal_docs/standards/PRD_拆分與交付單元規則.md
- PRD 不寫整個專案，一份 PRD 僅涵蓋一個主程式＋其子程式
- 主程式架構先建立，供工程師取得；PRD 分開撰寫、分開交付
- 主檔、參數、選項：全域用獨立主程式；單一主程式用則納入該主程式 PRD
- 系統設計建議與需求與情境流程審核建立歸檔時，須主動提醒：採用拆分最小邏輯有助於分給工程團隊平行開發
```

---

## 4. AI 系統設計建議時的要求

當 AI 提供**系統設計建議**或**需求規劃建議**時，須主動納入以下提醒（見 [PRD 拆分與交付單元規則](standards/PRD_拆分與交付單元規則.md) §7）：

> **建議採用拆分最小邏輯**：需求規劃可依主程式＋子程式為一交付單元，有助於分給工程團隊**平行開發**、分開交付、縮短整體時程。需求與情境流程審核建立歸檔時，建議依此邏輯劃分。
