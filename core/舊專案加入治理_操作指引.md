# 舊專案加入治理｜操作指引

本文件供**在既有專案 Repo 中加入文件治理**時使用。可將本文件提供給 AI，請其依步驟協助檢查與建立。  
**來源**：本 Template Repo（core/standards/templates/ops/index/_archive）；複製至舊專案後放置於該專案之 internal_docs/、00_Templates/ 等。

註：以下「來源路徑」指本 Template Repo；「舊專案放置」指專案 Repo 內之 internal_docs/、00_Templates/ 等。

---

## 使用方式

1. 將本文件複製至**舊專案** Repo（或提供給 AI 讀取）
2. 對 AI 下達指令，例如：「請依 `舊專案加入治理_操作指引.md` 協助我在本專案建立治理，並完成檢查清單」
3. AI 依本文件步驟執行，完成後回報檢查清單結果

---

## 二、替換清單（複製後須調整）

AI 複製檔案後，請在下列檔案中搜尋並替換：

| 搜尋 | 替換為 | 適用檔案 |
| :--- | :--- | :--- |
| `合群報關管理系統` | `[舊專案名稱]` | 協作與治理規範 |
| `HER_CHYUN`、`HER _CHYUN` | `[舊專案 Repo 名稱]` | 協作與治理規範、文件治理 |
| `需求方（您）`、`KyleShen` | `[需求方／開發方實際稱謂]` | 協作與治理規範 |
| `01_Discovery`、`02_Analysis` | 若舊專案有不同目錄名，改為對應路徑 | 協作與治理、文件治理 |
| `docs/` | 若舊專案對外文件放不同目錄，改為對應路徑 | 文件治理 |

**注意**：若舊專案無 `01_Discovery`、`02_Analysis`，可於治理規範中加註「本專案既有文件置於 [既有路徑]，新產出依本規範放置」。

## 三、建立步驟（AI 執行順序）

### Step 1：建立目錄結構

在舊專案 Repo 根目錄下建立：

```
internal_docs/
internal_docs/standards/    # 若需交付包相關標準
00_Templates/               # 若需範本
```

若 `docs/` 已存在則保留；若無則可建立（對外文件用）。

### Step 2：複製並調整治理規範

1. 複製 `協作與治理規範.md`、`docs_治理規範.md`、`程式編碼原則_共用規範.md` 至 `internal_docs/`
2. 依「二、替換清單」執行搜尋替換
3. 若舊專案有 `requirements/`、`specs/`、`design/` 等，在治理規範中加註對應關係（例如：「本專案之 02_Analysis 對應既有的 requirements/」）

### Step 3：建立 .cursorrules

在 Repo 根目錄建立或更新 `.cursorrules`，內容範本如下（請依舊專案調整）：

```markdown
# Role (角色設定)
你是協助本專案開發的 AI，依規格、雛型與驗收條件進行實作與檢核。

# 語言與格式
- 預設使用「臺灣繁體中文」
- 文件輸出為 Markdown (.md)
- 關鍵字使用 **粗體** 強調

# 治理與引用
- 文件產出依 internal_docs/協作與治理規範.md、internal_docs/docs_治理規範.md
- 對外文件放 docs/；內部治理放 internal_docs/
- 產出 PRD 時須含：模組與主程式代碼、權限矩陣、情境說明、欄位對照表（四項必備）
- 雛型檔名：YYYY-MM-DD_prototype_<程式代號>.html

# 命名與編碼
- 程式代號、TABLE、API 命名依「程式代號與權限矩陣」「欄位名稱對照表」
- 子程式識別採小寫；見 internal_docs/程式編碼原則_共用規範.md

# 權限與資安
- 依權限矩陣表實作，採最小權限原則
- 前後端皆須驗證身分與權限
- 密碼不明文儲存；敏感資料遮罩；HTTPS 傳輸
```

若舊專案已有 `.cursorrules`，請**合併**上述內容，勿覆蓋既有專案客製。

### Step 4：建立 internal_docs/index.md

建立 `internal_docs/index.md`，內容範本：

```markdown
# 內部治理文件

- [協作與治理規範](協作與治理規範.md)
- [文件治理規範](docs_治理規範.md)
- [程式編碼原則｜共用規範](程式編碼原則_共用規範.md)
```

### Step 5：（選用）複製範本與標準

若需產出 PRD 或交付開發包，自本 Template Repo 複製 `templates/` 與 `standards/` 下對應檔案至舊專案之 `00_Templates/` 與 `internal_docs/standards/`，並確認範本索引存在。

### Step 6：既有結構對應說明

若舊專案有不同目錄結構，在 `internal_docs/` 新增 `本專案目錄對應說明.md`，例如：

```markdown
# 本專案目錄對應說明

| 治理規範概念 | 本專案實際路徑 |
| :--- | :--- |
| 對外文件 | docs/ |
| 分析／PRD | requirements/ 或 02_Analysis/ |
| 雛型 | docs/prototype/ 或 design/prototype/ |
```

## 四、檢查清單（AI 完成後回報）

請 AI 依序檢查並回報：

| # | 檢查項 | 狀態（通過／略過／待補） |
| :---: | :--- | :---: |
| 1 | 已建立 `internal_docs/` 目錄 | |
| 2 | 已複製 `協作與治理規範.md` 至 internal_docs/ | |
| 3 | 已複製 `docs_治理規範.md` 至 internal_docs/ | |
| 4 | 已複製 `程式編碼原則_共用規範.md` 至 internal_docs/ | |
| 5 | 已依替換清單調整專案名稱、Repo、角色 | |
| 6 | 已建立或更新 `.cursorrules`，並引用上述規範 | |
| 7 | 已建立 `internal_docs/index.md` | |
| 8 | （選用）已建立 `00_Templates/` 並複製範本 | |
| 9 | （選用）已建立 `internal_docs/standards/` 並複製標準 | |
| 10 | 既有目錄與治理概念之對應已說明（若結構不同） | |

## 五、最小可行集合（僅 3 檔）

若僅欲快速加入治理，可只做：

1. **internal_docs/協作與治理規範.md** — 複製後替換專案名稱、Repo、角色
2. **internal_docs/docs_治理規範.md** — 複製後確認目錄對應
3. **.cursorrules** — 新建或合併，引用上述兩份規範

其餘可後續再補。

## 六、注意事項

| 項目 | 說明 |
| :--- | :--- |
| 不強制搬移 | 既有文件可維持原位；治理規範中加註對應即可 |
| 分支建議 | 建議開新分支（如 `feat/add-governance`）執行，經 PR 後合併 |
| 既有 .cursorrules | 若有，請合併內容，勿覆蓋 |
| 既有 .gitignore | 若有，確認 internal_docs 未被排除 |
