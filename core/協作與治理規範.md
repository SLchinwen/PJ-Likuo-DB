# 合群報關管理系統｜協作與治理規範

本文件定義專案**角色分工**、**產物歸屬**、**分支與合併規則**及**交付發布流程**，供需求方與開發方共同遵守。  
本專案之協作方式依 [公司專案標準規範｜需求與開發協作](公司專案標準規範_需求與開發協作.md) 執行，該規範適用於公司其他 e化 專案。

---

## 1. 角色與職責

| 角色 | 負責人 | 職責範圍 |
| :--- | :--- | :--- |
| **需求與業務** | 需求方（您） | 訪談、需求整理、PRD 撰寫、雛型設計與定稿、使用者測試與導入、上線後變更需求提出 |
| **開發與迭代** | 開發方（KyleShen） | 依 PRD 與雛型實作、單元／整合測試、程式迭代、配合變更修改與發布 |

**交付節奏**：需求方產出 PRD／雛型 → 開發方開發與測試迭代 → 交付 → 需求方與使用者測試導入 → 若有變更，雙方協同一次更改後發布。

### 1.1 與工程師之協作協議（現行）

- **需求方**：分享本 GitHub 專案（需求、PRD、雛型、文件）予工程方參考；**授權本 Repo** 給負責開發之人員，供其讀取。
- **工程方**：**公司開發 TEAM** 依需求方提供之資料，在**其自有之工程／開發專案**內建立專案管理與實作（採用 **VS** 等開發環境）；**不使用同一份 GitHub 專案**，以避免互相衝突。
- **每週同步**：需求方與公司開發 TEAM **每週**定期討論**開發進度**與**程式測試修正**，對齊待辦與下一週目標。
- **客戶回饋**：當**客戶有回饋**或需求調整時，需求方**通知**開發方；開發方於其專案內進行變更與差異分析；需求方 PRD／雛型更新仍以本 repo 為準，供工程方對照與同步。
- **文件版本**：**文件之版本與發布由需求方自行管理**；對外發布、客戶交付之文件與版次以需求方 repo 為依據。

## 2. 產物歸屬與工作邊界

上述內容若有變更，建議由需求方 commit 或由需求方 **Approve PR** 後再合併。

### 2.2 開發方主責（建議由開發方提交，需求方可審核）

- **03_Solution/**：規格細化、技術設計（若專案採用）

- **命名與格式**：遵守 `.cursorrules` 與本目錄下 [程式編碼原則](coding_standard_program_codes.md)（程式代號、雛型檔名等）。

### 2.4 各功能作業 PRD 產出要求

提供各功能作業之 PRD 或新文件需求時，須依下列要求產出，作為工程方**框架、TABLE、API 命名規劃**之基礎，並與客戶對齊表單與報表名稱：

| 項目 | 說明 |
| :--- | :--- |
| **模組與主程式代碼** | 先建立**完整**之模組代號與主程式代號清單（權限表用）；子程式、批次、API／TABLE 命名見 [程式編碼原則｜共用規範](程式編碼原則_共用規範.md) §4～§5。 |
| **程式 × 使用者角色權限矩陣表** | 依**情境**產出：主程式代號 × 使用者角色之權限矩陣表（可執行／可查詢／不可見等），標示各程式與角色之對應。 |
| **對應情境說明** | 各情境（如：承辦新增案件、主管審核、會計關帳、查詢歷史等）之操作者角色、可存取程式、與流程說明。 |
| **模組欄位名稱中英文對照表** | 該模組**所有客戶作業表單與報表**之欄位：**中文、英文、縮寫、用途**表列；給新文件需求時**同時建立或更新**，與客戶對齊表單與報表名稱，新欄位出現時提醒**新增／對應／修改／刪除**，避免需求失真。見 [文件治理規範](docs_治理規範.md) §6、範本 [tpl_欄位名稱對照表_交付用](../templates/tpl_欄位名稱對照表_交付用.md)。 |

上述 PRD 內容為**框架設計、資料表（TABLE）命名、API 命名與權限設計**之依據，需求方產出後供工程方於其專案內對應實作。

### 2.5 開發要求（含資安基本提醒）

開發方於**規劃與實作**時，除依 PRD 與雛型實作外，須注意下列**資安基本要求**，並依公司或客戶既有資安規範從嚴辦理：

| 類別 | 提醒要點 |
| :--- | :--- |
| **身分認證與登入** | 密碼不可明文儲存；登入／Session 逾時與登出機制；若客戶或公司要求，考量多因素認證（MFA）。 |
| **授權與權限** | 依 PRD 之**權限矩陣表**與情境實作，採**最小權限**原則；前後端皆須驗證身分與權限，不得僅依前端隱藏按鈕。 |
| **敏感資料** | 報關資料、個資、客戶資料之存取與儲存須受控；畫面與報表依角色遮罩或限制顯示；日誌與除錯輸出**不得**含密碼、完整個資或敏感欄位。 |
| **傳輸** | 對外或敏感連線使用 **HTTPS**；避免將敏感參數置於 URL 或 Cookie；API 與前後端通訊依環境要求加密。 |
| **日誌與稽核** | 重要操作（登入、權限變更、關鍵業務異動）可追溯；日誌內容避免記錄敏感資料；依公司或客戶規定保留與保護。 |
| **依循既有規範** | 依**公司資安政策**與**客戶資安／個資要求**（若有）規劃；若有弱掃、滲透測試或合規需求，預留時程與項目。 |

需求方得於專案啟動時與開發方確認上述項目之適用範圍；若有額外資安或合規需求，應寫入 PRD 或另附資安需求清單。

**雛型設計階段**：需求方或產出雛型者於**設計與生成雛型**時，請依 [文件治理規範](docs_治理規範.md) **§7 雛型設計與產出（含資安建議）** 之建議規範產出（權限區分示範、敏感欄位遮罩或假資料、登出／逾時示意、HTTPS 與後端驗證註記），使雛型與專案整體資安要求一致，並供開發實作時延續。

## 3. 分支與合併規則

### 3.1 分支約定

- **main**：正式版；對外發布與使用者測試導入皆以 main 為準。
- **develop**（選用）：整合用，開發方日常合併用；若未使用則以 main + feature 即可。
- **feature/xxx**、**fix/xxx**：功能或修正由開發方自 branch 開發，完成後以 **Pull Request** 合併至 main。

### 3.2 合併與審核

- **main 保護**：建議在 GitHub 設定 **Branch protection rule**，main 僅能透過 **Pull Request** 合併，不得直接 push。
- **誰來審核**：
  - 涉及 **01_Discovery、02_Analysis、docs、雛型** 的 PR：建議由**需求方**審核（Approve）後合併。
  - 純程式／設定／CI 的 PR：可由開發方合併，或由需求方視情況審核。
- **衝突**：由該 PR 發起人先 rebase/merge main 並解衝突，再請審核方審核。

### 3.3 需求方的 PRD／雛型更新

- main 已受保護時，**無法直接 push 到 main**，一律改為開 branch → PR → 審核 → 合併。
- 需求方變更（PRD、雛型、docs）建議開 `docs/xxx` 或 `update/xxx` 分支，完成後開 PR，由需求方自己 Approve 或由開發方協助合併。

### 3.4 開立分支與操作說明

因 **main 受保護**，所有變更都必須透過「開分支 → 推送分支 → 開 Pull Request → 審核 → 合併」才能進入 main。以下為標準流程與操作方式。

#### 3.4.1 分支命名建議

| 類型 | 命名範例 | 適用情境 |
| :--- | :--- | :--- |
| 文件／雛型更新 | `docs/update-prd-20260205`、`docs/雛型檢視連結` | 需求方更新 PRD、雛型、docs |
| 功能開發 | `feature/CUS110-案件建檔`、`feature/export-form` | 開發方新功能 |
| 錯誤修正 | `fix/聯絡單欄位`、`fix/build-error` | Bug 修正、小調整 |

#### 3.4.2 操作流程總覽

1. 在**本機**從最新 main 開新分支
2. 在該分支上修改、commit
3. **Push 分支**到遠端（不是 push main）
4. 到 **GitHub** 開 **Pull Request**（base: main，compare: 您的分支）
5. 審核（依規則需 1 人 Approve）→ 合併到 main
6. 本機切回 main 並 pull，再刪除已合併的本地分支（選用）

#### 3.4.3 使用 Cursor / VS Code 操作

1. **開新分支**  
   - 點左下角目前分支名稱（如 `main`）→ 選「建立新分支」或「Create new branch」  
   - 輸入分支名稱（例：`docs/update-prototype-20260205`）→  Enter

2. **確認已在該分支**  
   - 左下角應顯示新分支名稱，再進行編輯與存檔

3. **Commit**  
   - 在「變更」區寫 commit 訊息（例：`更新雛型檢視與協作規範`）  
   - 點「認可」或「Commit」

4. **推送到遠端**  
   - 點「同步變更」或「Push」  
   - 此時是 **push 目前分支**，不是 main，理應成功  
   - 若第一次 push 該分支，依提示選「Publish branch」或設定上游

5. **到 GitHub 開 PR**  
   - 開啟 `https://github.com/SLchinwen/HER_CHYUN`  
   - 若出現「Compare & pull request」橫幅，點進去；或到 **Pull requests** → **New pull request**  
   - **base** 選 `main`，**compare** 選您剛 push 的分支  
   - 填寫標題與描述 → **Create pull request**

6. **審核與合併**  
   - 若規則要求 1 人 Approve：由審核方點 **Review** → **Approve** → 再點 **Merge pull request**  
   - 合併後可選「Delete branch」刪除遠端分支（選用）

7. **本機更新 main（選用）**  
   - 在 Cursor 左下角切回 `main` → 點「同步變更」或 **Pull**，取得剛合併的內容  
   - 若不再需要該分支，可刪除本機分支（分支清單中右鍵 → 刪除）

#### 3.4.4 使用指令列操作

```bash
# 1. 確保在 main 且為最新
git checkout main
git pull origin main

# 2. 開新分支並切換過去
git checkout -b docs/update-prototype-20260205

# 3. 修改檔案後，加入並 commit
git add .
git commit -m "更新雛型檢視與協作規範"

# 4. 推送分支到遠端（不是 push main）
git push -u origin docs/update-prototype-20260205

# 5. 到 GitHub 網頁開 PR、審核、合併（無法用指令完成）

# 6. 合併後，本機切回 main 並更新
git checkout main
git pull origin main
git branch -d docs/update-prototype-20260205   # 刪除已合併的本地分支（選用）
```

#### 3.4.5 常見錯誤與處理

| 狀況 | 原因 | 處理方式 |
| :--- | :--- | :--- |
| 推送時出現 **GH013**、**Changes must be made through a pull request** | 嘗試直接 push 到 main，被分支保護拒絕 | 勿直接 push main；改為開分支 → push 分支 → 到 GitHub 開 PR 合併 |
| 同步變更時一直失敗 | IDE 預設可能對應到 main，直接 push 被拒 | 確認左下角目前分支為您的工作分支（如 `docs/xxx`），再按「同步變更」 |
| PR 無法合併，提示需要審核 | 規則要求至少 1 人 Approve | 由審核方在 PR 頁面點 **Review** → **Approve**，即可出現 **Merge** 按鈕 |

## 4. 交付與發布流程

### 4.1 開發交付

1. 開發方完成當輪實作與測試後，以 **PR 合併至 main**（或先合併至 develop 再 PR 至 main，依 3.1 約定）。
2. 需求方審核 PR（尤其涉及需求／雛型對齊），合併後即視為**當輪交付**。

### 4.2 使用者測試導入

1. 需求方以 **main** 為基準，與使用者進行測試與導入（文件、雛型或系統依專案階段）。
2. 發布管道依現有設定（如 GitHub Pages、MkDocs）從 main 建置／部署。

### 4.3 變更後協同發布

1. 使用者測試後若有**變更需求**，由需求方整理（必要時更新 PRD／雛型或 issue）。
2. 開發方依變更內容在 **feature/fix branch** 修改，完成後開 PR 至 main。
3. 需求方審核 PR，合併後由需求方或開發方依共識**觸發發布**（如 push 觸發 GitHub Actions）。
4. 發布完成後，需求方再與使用者確認上線／導入結果。

## 6. 工作風險與因應

| 風險 | 說明 | 建議因應 |
| :--- | :--- | :--- |
| **需求與實作不同步** | PRD／雛型更新後開發方未即時得知，或實作偏離需求 | 變更 PRD／雛型後通知開發方；PR 描述註明對應的 PRD／雛型段落或 issue；重要需求以 issue 追蹤 |
| **交付定義不清** | 「交付」與「可上線」標準不明，易產生認知落差 | 在 PRD 或協作規範中約定「交付」定義（例如：通過驗收清單、文件更新完成）；每輪交付前簡短對齊 |
| **main 未保護** | 若未設 Branch protection，有人直接 push 到 main，繞過審核 | 盡快設定 main 的 Branch protection（需 PR 才能合併）；必要時勾選「不允許 bypass」 |
| **審核延遲** | 需求方未即時審 PR，開發方卡關或改動堆積 | 約定審核時限（如 1～2 個工作天）；或約定緊急時可先合併再補審，並在 PR 註記 |
| **版本對不回來** | 使用者測試時發現問題，無法對應「哪一版」出錯 | main 重要節點打 **Tag**（如 `v0.1.0`、`delivery-20260205`）；發布紀錄註明對應 Tag 或 commit |
| **單點依賴** | 需求方或開發方其一請假／離職，知識與決策集中在一人 | 關鍵決策與驗收條件寫進 PRD／文件；重要變更在 PR／issue 留紀錄；必要時交叉了解對方產物 |
| **變更頻繁與範圍蔓延** | 使用者測試後變更多且散，開發反覆改、難收斂 | 變更集中成一批、寫成簡短「變更清單」或 issue，再開一輪開發；約定每輪「凍結」時間與發布節奏 |
| **CODEOWNERS 僅建議** | GitHub CODEOWNERS 只會「建議」reviewer，不強制審核 | 在 Branch protection 中啟用「Require review from Code Owners」，並至少 1 人 Approve 才能合併（需 GitHub 付費方案則視授權而定） |

> 若為 GitHub Free：無法強制 Code Owners 審核，可改為**約定**「動到 02_Analysis、docs 的 PR 一定由需求方 Approve 後才合併」，並在 PR 標題或描述標註 `[需需求方審核]`。
