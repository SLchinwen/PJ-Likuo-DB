# 公司專案標準規範｜需求與開發協作

本規範為**公司 e化 專案**之標準協作方法與步驟，適用於**需求方持有 Repo、公司開發 TEAM 負責實作**之各專案。新專案可依本規範建立協作方式與 PRD 產出要求，保持公司內一致之品質與可追溯性。

**文件治理辦法與規則**：細化之協作、文件治理、程式與雛型規範、交付與 AI 產出等，見 [readme_internal_docs](readme_internal_docs.md)（總覽）與 [index_internal_docs](index_internal_docs.md)（分類索引）、[index/00_overview.md](../index/00_overview.md)；各專案可引用本 repo 為中央標準庫。

---

## 1. 適用範圍

| 項目 | 說明 |
| :--- | :--- |
| **適用情境** | 需求方（業務／專案負責人）主導需求與文件，**授權其 Repo** 給開發人員；**開發由公司開發 TEAM** 執行，採用 **VS（Visual Studio）** 等開發環境；雙方**不使用同一份 GitHub 專案**，以避免衝突。 |
| **適用專案** | 各 e化 專案、報關管理、及其他依相同模式委由公司開發 TEAM 實作之專案。 |
| **不適用** | 開發與需求為同一團隊且共用同一 repo 之專案；或公司另有約定之協作模式。 |

---

## 2. 協作方法摘要

| 角色 | 作法 |
| :--- | :--- |
| **需求方** | 持有並維護**需求 Repo**（需求、PRD、雛型、文件）；**授權該 Repo** 給負責開發之人員；文件版本與對外發布**自行管理**。 |
| **開發方** | **公司開發 TEAM** 依需求方提供之資料，在**自有之工程／開發專案**內建立專案管理與實作（如採用 **VS** 開發）；**不使用需求方同一份 GitHub**，避免互相衝突。 |
| **同步與迭代** | 需求變更或客戶回饋時，**需求方通知調整**；開發方在其專案內進行變更與差異分析，並以需求方 Repo 之 PRD／雛型為對照依據。 |

---

## 3. 標準步驟（各專案可依此執行）

### 3.1 需求產出階段

1. **訪談與需求整理**：需求方完成訪談、需求展開、客戶資料卡等（置於需求方 Repo，如 `01_Discovery/`）。
2. **模組與主程式代碼**：先建立**完整**之模組代號與主程式代號清單（權限表用）；見 [程式編碼原則｜共用規範](程式編碼原則_共用規範.md)。
3. **PRD 產出（四項必備）**：  
   - 模組與主程式代碼（已建立）  
   - 程式 × 使用者角色權限矩陣表  
   - 對應情境說明  
   - **模組欄位名稱中英文對照表**（該模組所有客戶作業表單之欄位：中文、英文、縮寫、用途；給新文件需求時同時建立或更新，與客戶對齊表單與報表名稱，新欄位時提醒新增／對應／修改／刪除，避免需求失真）  
   見 [文件治理規範](docs_治理規範.md) §6 標準分析產出要求、[協作與治理規範](協作與治理規範.md) §2.4。
4. **雛型與對外文件**：雛型（如 HTML）、專案總攬、程式清單等置於需求方 Repo（如 `02_Analysis/`、`docs/`）。

### 3.2 授權與開發啟動

5. **授權 Repo**：需求方將**需求 Repo 之存取權**授權給負責該專案之開發人員（或公司開發 TEAM 代表），供其讀取 PRD、雛型、文件。
6. **開發方建立自有專案**：公司開發 TEAM 在**自有之版本庫／專案**內建立工程專案（如採用 **VS** 開發），依 PRD 與雛型進行實作與專案管理；**不直接於需求方 Repo 提交程式碼**，避免衝突與權限混雜。
7. **對齊基準**：開發方以需求方 Repo 之**當時版次**（或約定之 Tag／commit）為需求基準，框架、TABLE、API 命名依 PRD 之模組與主程式代碼、權限矩陣、情境說明規劃。

### 3.3 開發進行與定期同步

8. **每週定期討論**：需求方與公司開發 TEAM **每週**進行會議，討論**開發進度**與**程式測試修正**，對齊待辦、阻塞項與下一週目標。
9. **測試與修正**：開發方於其專案內進行單元／整合測試與修正；需求方依需要提供驗收情境或測試案例（可寫入 PRD 或另附）。

### 3.4 變更與客戶回饋

10. **客戶回饋或需求調整**：當**客戶有回饋**或需求方決定調整時，**需求方通知開發方**（如郵件、群組、會議），並在需求方 Repo 更新 PRD／雛型／文件，必要時註明版次或變更摘要。
11. **開發方迭代**：開發方於**其專案**內進行變更與差異分析，完成後依約定方式交付或部署；需求方 Repo 仍為**文件與對外發布之單一依據**。

### 3.5 交付與發布

12. **交付定義**：依專案約定之「交付」定義（如通過驗收清單、文件已更新）進行驗收；對外發布與客戶交付之**文件版次以需求方 Repo 為準**。
13. **發布管理**：文件版本與發布由**需求方自行管理**；若由開發方負責部署或建置，應以需求方指定之版次或檔案為依據。

## 4. 實際運作要點（公司開發 TEAM）

| 項目 | 說明 |
| :--- | :--- |
| **授權** | 需求方授權其 Repo 給負責開發之人員，開發人員**唯讀**需求與文件即可；不需在需求方 Repo 提交程式碼。 |
| **開發環境** | 公司開發 TEAM 採用 **VS（Visual Studio）** 等工具，在**自有專案**內開發、測試、版控。 |
| **每週同步** | **每週**與需求方定期討論**開發進度**與**程式測試修正**，即時對齊問題與排程。 |
| **客戶回饋** | 當需求方收到**客戶回饋**並決定調整時，會**通知**開發方；開發方於自有專案內進行調整與差異分析。 |
| **命名與權限** | 框架、TABLE、API 命名及權限設計，依需求方 PRD 之**模組與主程式代碼、權限矩陣表、情境說明**為準。 |
| **資安基本要求** | 規劃與實作時須注意**資安基本要求**（身分認證、授權與最小權限、敏感資料保護、傳輸加密、日誌與稽核、依公司或客戶資安規範）；詳見各專案 [協作與治理規範](協作與治理規範.md) §2.5 開發要求。 |

## 5. 其他專案如何套用本規範

| 步驟 | 作法 |
| :--- | :--- |
| 1. 新專案立項 | 確認該專案為「需求方持有 Repo、公司開發 TEAM 實作、不同 repo」模式，即適用本規範。 |
| 2. 建立需求 Repo（或沿用既有） | 需求方建立或指定一 Repo 存放需求、PRD、雛型、對外文件；目錄結構可參考專案 Repo 建議（01_Discovery、02_Analysis、docs、internal_docs、00_Templates）。 |
| 3. 複製或引用公司標準文件 | 將本規範、[程式編碼原則｜共用規範](程式編碼原則_共用規範.md)、[文件治理規範](docs_治理規範.md) §6、[協作與治理規範](協作與治理規範.md) 複製至新專案 Repo 之 internal_docs，或以連結引用；專案專屬之協作細節（如專案名稱、Repo 網址、會議頻率）可寫入該專案之「協作與治理規範」或專案說明。 |
| 4. PRD 依標準產出要求產出 | 各功能作業 PRD 須含：模組與主程式代碼、權限矩陣表、情境說明、**模組欄位名稱中英文對照表**（四項必備）；範本見本 Template Repo `templates/` 下對應範本（如 tpl_程式代號與權限矩陣_交付用.md、tpl_欄位名稱對照表_交付用.md 等）。 |
| 5. 授權與每週同步、客戶回饋通知 | 需求方授權 Repo 給該專案負責開發人員；與公司開發 TEAM 約定每週討論與客戶回饋時之通知方式。 |

## 6. 此方式之優缺點與建議配套措施

### 6.1 優點

| 優點 | 說明 |
| :--- | :--- |
| **權責與版本清楚** | 需求方單一掌握需求與文件，對外發布與客戶交付口徑一致；開發方專注實作與測試，不需處理需求 Repo 之 commit／分支權限。 |
| **避免協作衝突** | 不同 repo 下，不會發生需求文件與程式碼互相覆蓋、merge 衝突或審核流程混雜。 |
| **開發環境獨立** | 公司開發 TEAM 可依既有習慣使用 VS 與自有版控、CI/CD，不受需求方 Repo 工具或流程限制。 |
| **每週同步可即時對齊** | 定期討論開發進度與程式測試修正，問題與排程可即時對齊，減少認知落差。 |
| **客戶回饋有單一入口** | 客戶回饋由需求方收斂後通知調整，避免客戶直接對開發方造成多源頭變更。 |
| **需求產出有結構** | PRD 三項必備（代碼、權限矩陣、情境）提供開發方明確的命名與權限設計依據，利於框架、TABLE、API 對齊。 |
| **可複用至其他專案** | 同一套方法與步驟、程式編碼與範本可套用於公司其他 e化 專案，維持一致品質。 |

### 6.2 缺點與風險

| 缺點／風險 | 說明 |
| :--- | :--- |
| **雙邊同步依賴人為** | 需求方更新 PRD／雛型後若未主動通知或標註版次，開發方可能沿用舊版，導致實作與需求不一致。 |
| **差異分析產出未約定** | 開發方在其專案內進行變更與差異分析時，若未約定產出格式（如變更清單、影響對照表），需求方難以驗收與追蹤。 |
| **需求方為文件單點** | 文件與發布集中需求方，若人力或交接不順，可能影響開發方取得最新規格。 |
| **無共同工單追蹤** | 需求變更與 bug 若分散在需求方與開發方不同工具，較難單一視圖追蹤「需求－實作－驗收」對應。 |
| **版次對照需手動維護** | 開發方專案之某版對應需求方 Repo 哪一版次，需雙方約定方式標註，否則事後難以對回。 |

### 6.3 建議配套措施

以下配套可降低上述風險，建議新專案啟動或既有專案檢討時一併約定。

#### 同步與通知

| 配套 | 作法 |
| :--- | :--- |
| **需求更新通知** | 需求方更新 PRD、雛型或程式代號後，以約定管道（如 Email、群組、專案頻道）**通知**開發方，並註明**版次或變更摘要**（可附簡短變更清單）。 |
| **版次對照** | 需求方在對外或內部文件維持「PRD／雛型版次與發布日」表；開發方於其專案標註「本版對應需求方 Repo 之版次／Tag／commit」，便於事後對回。 |

#### 變更與差異分析

| 配套 | 作法 |
| :--- | :--- |
| **差異分析產出格式** | 約定：迭代或客戶回饋調整時，開發方提供簡要**變更／差異清單**（如影響之程式、TABLE、API 或畫面），供需求方驗收與留檔。 |
| **需求變更清單範本** | 需求方提供「需求變更清單」範本（變更項目、影響程式代號、預期行為）；每次客戶回饋或需求調整時填寫並交開發方，利於估價與差異分析。 |

#### 交付與驗收

| 配套 | 作法 |
| :--- | :--- |
| **交付定義寫入規範** | 在專案之協作與治理規範中明列「一輪交付」定義（例如：通過驗收清單、文件已更新、權限矩陣與情境已驗證），減少認知落差。 |
| **權限與情境納入驗收** | 上線前將「權限矩陣表」與「情境說明」納入驗收項目，確認角色與程式權限、情境流程與實作一致。 |

#### 節奏與範圍

| 配套 | 作法 |
| :--- | :--- |
| **鎖定階段範圍** | 明訂第一階段開發範圍（如單一模組或單一作業），其餘待 PRD 完成後再排程，避免範圍蔓延與需求未齊即開發。 |
| **每週會議議程** | 每週討論固定涵蓋：本週進度、測試結果與待修正項、下一週目標、阻塞項與需求疑義；必要時紀錄簡要會議紀要並存於需求方 Repo 或雙方可存取處。 |

#### 知識與交接

| 配套 | 作法 |
| :--- | :--- |
| **關鍵決策留檔** | 主程式代號取捨、權限矩陣定案、重大需求變更等寫入需求方 Repo 之 PRD 或 internal_docs，降低人員異動後銜接成本。 |
| **新專案檢查清單** | 新專案啟動時依本規範 §5 檢查：授權、PRD 三項必備、每週同步與客戶回饋通知方式是否已約定。 |
